## L1 — Capability (CTO view)

Projects are eRPC's multi-tenancy unit: a single eRPC process serves any number of fully isolated tenants ("projects"), each with its own upstream pool, network registry, health/score tracker, selection-policy engine, consumer authentication, and an optional project-wide rate-limit budget. A request is routed to a project via the URL path (`/<projectId>/<architecture>/<chainId>`), domain aliasing rules, or gRPC metadata, then passes through per-project CORS, method allow/ignore filters, auth strategies, and rate limiting before any upstream is touched. This lets one deployment safely serve e.g. a public frontend (CORS-restricted, IP rate-limited) and internal indexers (secret-token auth, unlimited) with different failover policies, without cross-tenant interference in metrics or upstream scoring.

## L2 — Mechanics (staff-engineer view)

**Isolation boundaries.** `ProjectsRegistry` (erpc/projects_registry.go:52-70) is built once at startup by `NewERPC` (erpc/erpc.go:68-78) and eagerly registers every `projects[]` entry from config (erpc/projects_registry.go:96-101). `RegisterProject` (erpc/projects_registry.go:123-218) creates, **per project**: a `health.Tracker` (score metrics, window = project `scoreMetricsWindowSize` or the package fallback `ScoreMetricsWindowSize` = 1 minute, erpc/projects_registry.go:50,133-137), a `thirdparty.ProvidersRegistry`, an `upstream.UpstreamsRegistry`, an optional `auth.AuthRegistry` (only when `auth` is configured, erpc/projects_registry.go:186-192), a `policy.Engine` (selection policy JS runtime, erpc/projects_registry.go:195-202), and a `NetworksRegistry` (erpc/projects_registry.go:203-212). **Shared across projects**: the rate-limiters registry (budgets are global, referenced by id), shared-state registry, the EVM JSON-RPC cache (scoped per project at network-prepare time via `evmJsonRpcCache.WithProjectId(projectId)`, erpc/networks_registry.go:316-318), the vendors registry, the proxy-pool registry, and the compiled TS-config user script (erpc/projects_registry.go:72-94). Duplicate project ids fail startup with `ErrProjectAlreadyExists` (erpc/projects_registry.go:124-126). `ProjectsRegistry.Bootstrap` calls each project's `Bootstrap`, which boots the upstreams registry then the networks registry (erpc/projects_registry.go:106-110, erpc/projects.go:36-39); this happens before the HTTP server starts (erpc/init.go:99-112).

**Request routing to a project (HTTP).** `handleRequest` (erpc/http_server.go:226-447) first applies **domain aliasing**: each `server.aliasing.rules[]` entry is matched against the request `Host` (port stripped) with `common.WildcardMatch`; the first match pre-selects `projectId`/`architecture`/`chainId` (erpc/http_server.go:242-257). Then `parseUrlPath` (erpc/http_server.go:810-1006) fills the remaining values from path segments — a 5-way case analysis over which values were preselected (see L3). `/admin` (POST/OPTIONS only) flags the admin path; a trailing `/healthcheck` segment, or any non-POST/non-OPTIONS method, flags healthcheck (erpc/http_server.go:836-849,1001-1003). Project resolution happens via `erpc.GetProject` — unknown id returns `ErrProjectNotFound` → HTTP 404 (erpc/projects_registry.go:112-121, common/errors.go:574-589, erpc/http_server.go:1304-1305). On gRPC the project comes from `x-erpc-project` metadata (required), chain from `x-erpc-chain-id`, architecture from `x-erpc-architecture` defaulting to `"evm"` (erpc/grpc_server.go:133-158); the shared `RequestProcessor` then performs the same auth + forward flow (erpc/request_processor.go:35-67).

**CORS.** Two CORS surfaces exist: admin (`admin.cors`, checked before admin request handling, erpc/http_server.go:295-301) and per-project (`projects[].cors`, checked right after project resolution, erpc/http_server.go:343-347). `handleCORS` (erpc/http_server.go:1008-1077) is deliberately permissive: if the request carries **no `Origin` header it is always allowed** (no point enforcing CORS for non-browser clients — the code comment explicitly states CORS here only deters casual frontend embedding, not real attackers, erpc/http_server.go:1010-1018). With an Origin present, each `allowedOrigins` pattern is evaluated with `WildcardMatch` (glob + boolean grammar). Allowed origin → echo the origin in `Access-Control-Allow-Origin` plus methods/headers/expose/credentials/max-age headers; OPTIONS preflight then terminates with 204. Disallowed origin → **no CORS headers but the request still proceeds** (browser blocks it; curl etc. succeed), except OPTIONS which gets a bare 204 (erpc/http_server.go:1035-1051). Note CORS for projects runs only when `cors` is configured (nil → OPTIONS falls through to the JSON-RPC body parser).

**Per-request pipeline inside a project.** Each entry of a (possibly batch) request body is processed in its own goroutine (erpc/http_server.go:440-675): payload validation → `forwardHeaders` capture (wildcard match against incoming header names; matched values are forwarded to upstream HTTP requests, erpc/http_server.go:478-494, clients/http_json_rpc_client.go:735-740) → `ignoreMethods`/`allowMethods` gate (allow overrides ignore; rejected methods return JSON-RPC error `-32601` without touching auth or upstreams, erpc/http_server.go:499-550) → auth payload extraction from query/headers (auth/http.go:13-88) → `project.AuthenticateConsumer` (no-op returning nil user when project has no auth config, erpc/projects.go:94-99) → network id resolution (from path or `"networkId"` field in the body; lazy network bootstrap on first use) → directive defaults + HTTP enrichment (User-Agent mode per project) → `project.Forward`.

**`PreparedProject.Forward`** (erpc/projects.go:101-254): resolves the network (lazy-loads on first request; lazily created network configs are exposed back into `project.Config.Networks` via `ExposeNetworkConfig`, erpc/projects.go:60-77), sets it on the request **before** rate limiting so metrics have the network label, then `AcquireRateLimitPermit` enforces the project-level budget (erpc/projects.go:111-116). After that, the "requests received" counter fires (so project-rate-limited requests are *not* counted as received), then `doForward` runs: for EVM, a project-level pre-forward hook may short-circuit entirely (e.g. `eth_chainId` served statically, cache-affecting normalizations for `eth_blockNumber`/`eth_call`/`eth_getLogs`/`trace_filter`, architecture/evm/hooks.go:10-36), otherwise `network.Forward` plus a network post-forward hook (erpc/projects.go:256-268). On success/failure the project emits the canonical request metrics with user/agent labels and logs; if the network has shadow upstreams, the response is cloned and mirrored asynchronously for comparison (erpc/projects.go:137-164, erpc/shadow.go:16-266).

**Rate-limit layering relevant to projects.** Project-level: `projects[].rateLimitBudget` names a budget in global `rateLimiters.budgets[]`; `AcquireRateLimitPermit` looks up rules matching the method and calls `TryAcquirePermit(... origin="project")`; over-limit → `ErrProjectRateLimitRuleExceeded` → HTTP 429 (erpc/projects.go:270-310, common/errors.go:1738-1758). Auth-level: each auth strategy may carry its own `rateLimitBudget`, overridable per-user (`User.RateLimitBudget` from JWT claim / database record / strategy config); checked inside `Authenticate` after a strategy succeeds, over-limit → `ErrAuthRateLimitRuleExceeded` → 429 (auth/authorizer.go:117-157, auth/registry.go:77-80, common/errors.go:545-568). Network- and upstream-level budgets are separate layers (see rate-limiter KB). All layers **fail open**: no store configured, Redis not yet connected, Redis timeout, or admission-semaphore saturation all allow the request (upstream/ratelimiter_budget.go:156-158,248-252,291-304,349-461; upstream/ratelimiter_registry.go:58-67).

**Failure modes.** Project not found → 404 before any CORS headers for that project can be emitted. Budget referenced but undefined → startup validation error (common/validation.go:640-644). Auth configured but no strategy matches the payload → 401 `ErrAuthUnauthorized` with joined strategy errors (auth/registry.go:86-95). Rate-limit store down → fail-open (availability over enforcement, by design).

**Legacy field translation mechanics.** `ProjectConfig.UnmarshalYAML` (common/config.go:563-579) inline-captures all seven legacy fields from YAML alongside the canonical schema. If any are non-zero they are stashed in `ProjectConfig.LegacyProject *LegacyProjectFields` (never serialized: `yaml:"-" json:"-"`). The `LoadConfig` pipeline calls `common.LegacyTranslateFn` = `legacy.TranslateFromConfig` (common/legacy/translate.go:212-243) **before** `SetDefaults` and `Validate`. The translator classifies fields into "semantic" (alters selection behavior: `routingStrategy`, `scoreSwitchHysteresis`, `scoreMinSwitchInterval`) and "inert" (warning-only: `scoreGranularity`, `scorePenaltyDecayRate`, `scoreMetricsMode`, `scoreRefreshInterval`). Inert fields only: warnings emitted, stash cleared, canonical default policy installs (common/legacy/translate.go:59-68). Semantic fields present: per-network `selectionPolicy.evalFunc` synthesized UNLESS the network already has an explicit `selectionPolicy.eval` / `selectionPolicy.evalFunc` (modern fields always win, common/legacy/translate.go:80-83). After translation, `clearLegacyStashes` nils `LegacyProject` on every project (common/legacy/translate.go:318-328). **Migration guidance**: remove all seven legacy fields and write an explicit `selectionPolicy.eval` using the chainable stdlib (`sortByScore`, `stickyPrimary`, `rotateBy`, etc.); see https://docs.erpc.cloud/migration/selection-policy. The synthesized JS can be viewed by temporarily enabling debug logging and observing `selectionPolicy.evalFunc` in the admin `erpc_project` response (the field is populated after `SetDefaults` compiles it).

### Behaviors & algorithms

**URL path → (project, architecture, chain) resolution** (`parseUrlPath`, erpc/http_server.go:810-1006). Path is `path.Clean`ed and split; preselected values come from aliasing. Decision matrix (P=project, A=architecture, C=chain preselected):

- none preselected: 1 seg → P; 2 segs → P + (network-alias resolve of seg2, else A); 3 segs → P/A/C; 0 segs → only valid for healthcheck; >3 → error `must only provide /<project>/<architecture>/<chainId>` (erpc/http_server.go:853-877).
- P preselected: 1 seg → alias resolve else A; 2 segs → A/C; 3 segs → full explicit override; (erpc/http_server.go:879-904).
- P+A preselected: 1 seg → alias resolve (must yield both) else C; 3 segs → full override (erpc/http_server.go:906-935).
- P+A+C preselected: only `/` (or healthcheck) allowed; >1 seg → error (erpc/http_server.go:937-941).
- A+C preselected: 1 seg → P; 3 segs → full override (erpc/http_server.go:943-961).
- A only preselected: 1 seg → P; 2 segs → P/C; 3 segs → full override; 0 segs allowed for global healthcheck (erpc/http_server.go:963-983).
- P+C without A → error (erpc/http_server.go:985-988).
Post-checks: project required unless healthcheck (erpc/http_server.go:993-995); if A or C present, A must be `"evm"` (`architecture is not valid (must be 'evm')`, erpc/http_server.go:997-999, common/network.go:35-37); **any non-POST/non-OPTIONS method becomes a healthcheck** (erpc/http_server.go:1001-1003). `/admin` only with exactly one segment + POST/OPTIONS (erpc/http_server.go:836-838). Full matrix encoded in TestHttpServer_ParseUrlPath (erpc/http_server_test.go:3378-3661).

**Network alias resolution.** Aliases (`projects[].networks[].alias`) are registered eagerly for static networks at registry construction (erpc/networks_registry.go:67-79) and lazily when a network is prepared (erpc/networks_registry.go:322-328). `ResolveAlias(alias) → (architecture, chainId)` or `("","")` (erpc/networks_registry.go:247-255). Duplicate alias with a different target keeps the first and logs a warning (erpc/networks_registry.go:334-348). Network ids must look like `evm:<int>` (`util.IsValidNetworkId`, util/ids.go:21-27); alias misses in path-position fall through to being treated as an architecture, producing `architecture is not valid` for bogus aliases. If both path and body lack a network, the body's `"networkId"` field (e.g. `"evm:42161"`) is consulted (erpc/http_server.go:613-634); otherwise: `architecture and chain must be provided in URL ... or in request body ... or configureed via domain aliasing` (sic, erpc/http_server.go:636-642). Alias behavior verified in TestProject_NetworkAlias (erpc/projects_test.go:494-589).

**Wildcard/boolean pattern grammar** (used by CORS origins, aliasing domains, forwardHeaders, ignore/allowMethods, rate-limit rule methods; common/matcher.go:34-47,221-272,286-396): tokens `|` (OR), `&` (AND), `!` (NOT), parentheses, space = token separator; leaf patterns are glob matches via `github.com/IGLOU-EU/go-wildcard/v2` `wildcard.Match` where `*` = any sequence, `?` = exactly one char, **`.` = any single char** (library source wildcard_match.go: "case '.': It matches any single character"). `<empty>` matches the empty string. For values starting with `0x`, prefixes `>=`, `<=`, `>`, `<`, `=` enable hex/decimal numeric comparison (common/matcher.go:372-396). Operator precedence: NOT > AND > OR.

**CORS algorithm** (`handleCORS`, erpc/http_server.go:1008-1077) — returns "continue?":
1. No `Origin` header → return true (no headers, no metrics).
2. Increment `erpc_cors_requests_total`.
3. `allowed = any(WildcardMatch(pattern, origin))` over `allowedOrigins`.
4. Disallowed: increment `erpc_cors_disallowed_origin_total`; OPTIONS → write 204 + return false; else → return true **without any Access-Control-* headers** (request proceeds; test: erpc/http_server_test.go:3332-3376 asserts 204 + empty headers).
5. Allowed: set `Access-Control-Allow-Origin: <origin>` (echo, never `*`), `-Allow-Methods`, `-Allow-Headers`, `-Expose-Headers` (joined `", "`), `-Allow-Credentials` only if true, `-Max-Age` only if >0.
6. OPTIONS → increment `erpc_cors_preflight_requests_total`, write 204, return false (caller also returns for any OPTIONS after a true CORS result, erpc/http_server.go:297,344). Tests: erpc/http_server_test.go:3239-3330.

**Method gating order** (per sub-request): validate JSON-RPC → forwardHeaders capture → ignoreMethods → allowMethods → unsupported-method short-circuit → auth → admin-or-project dispatch → network resolve → forward (erpc/http_server.go:460-675). The unsupported-method response body is `{"jsonrpc":"2.0","id":<echo>,"error":{"code":-32601,"message":"method not supported: X"}}` — but due to an inverted error check (`if jrr, err := nq.JsonRpcRequest(); err != nil { ... }` erpc/http_server.go:534-537) the request id is echoed only on the parse-failure branch; in practice `id` is `null` and version is the default `"2.0"`.

**Auth payload extraction precedence (HTTP)** (auth/http.go:13-88): `?token=` (deprecated) → `?secret=` → header `X-ERPC-Secret-Token` → header `Authorization: Basic …` (password half = secret; username ignored) → `Authorization: Bearer …` (JWT) → `?jwt=` → `?signature=&message=` (SIWE; message may be base64) → headers `X-Siwe-Message` + `X-Siwe-Signature` → fallback type `network` (client-IP based). The same payload type drives which strategies `Supports()` it.

**Consumer authentication** (`AuthRegistry.Authenticate`, auth/registry.go:46-96): nil payload → 401; zero strategies → allow (nil user). Iterate strategies in order; skip non-applicable (method filter / unsupported payload type); first success attaches `User` to the request (so `user` metric labels work) and enforces the strategy's rate-limit budget — note a rate-limit failure returns `(user, err429)`. If every applicable strategy errors: single error returned as-is, multiple joined into one `ErrAuthUnauthorized`; if none applicable: 401 "no auth strategy matched make sure correct headers or query strings are provided". `User` = `{Id, RateLimitBudget}` (common/user.go:3-6); `req.UserId()` returns `"n/a"` when unauthenticated (common/request.go:1271-1282).

**Project-level rate limiting** (`AcquireRateLimitPermit`, erpc/projects.go:270-310): no budget configured → allow. Budget id resolved via shared registry (`ErrRateLimitBudgetNotFound` if missing at runtime — normally impossible because validation runs at startup). Rules filtered by method; if ≥1 rule, `TryAcquirePermit(projectId, req, method, vendor="", upstreamId="", authLabel="", origin="project")`. Envoy descriptor entries: always `{method}`, plus `{ip}`/`{user}`/`{network}` per rule scope flags; domain = budget id (upstream/ratelimiter_budget.go:248-277). Multiple matching rules are evaluated in parallel; ANY over-limit blocks (upstream/ratelimiter_budget.go:207-243). Deny → `ErrProjectRateLimitRuleExceeded{project, budget, rule:"method:<m>"}` → HTTP 429 (common/errors.go:1738-1758, erpc/http_server.go:1306-1312). Behavior verified by TestProject_Forward/ForwardCorrectlyRateLimitedOnProjectLevel — 5 calls against maxCount 3/minute, last error is `ErrProjectRateLimitRuleExceeded` (erpc/projects_test.go:22-130). Note `GetRulesByMethod` may double-count a rule when `rule.Config.Method == method` AND wildcard also matches — harmless (same limit twice; upstream/ratelimiter_budget.go:63-78).

**`PreparedProject.Forward` metric/label mechanics** (erpc/projects.go:101-254): `network.Label()` = alias if set else networkId (erpc/networks.go:278-286). Success: vendor/upstream labels are `"<cache>"`/`"<cache>"` for cache hits, `"n/a"` when upstream missing; emits successful-request counter (incl. `attempt`, `finality`, `emptyish`, `user`, `agent_name`), duration histogram, and block-range heatmap when a block number is extractable (erpc/projects.go:176-220, erpc/block_heatmap.go:16-60). Failure: failed-request counter with `error` = `common.ErrorFingerprint(err)`, `severity` = classified severity; duration histogram with vendor/upstream = `"<error>"` (erpc/projects.go:221-251). Client errors and `ErrCodeEndpointExecutionException` are logged at info as "client-side exception" without request dump (erpc/projects.go:222-224).

**Shadow traffic** (project layer; erpc/projects.go:137-164, erpc/shadow.go:16-266): when the network reports shadow upstreams, the JSON-RPC response is cloned (with upstream/cache/attempt metadata) and compared asynchronously against each shadow upstream (per-upstream `shadow.sampleRate` sampling, `shadow.ignoreFields[method]` hash exclusions, emptyish-equivalence). Outcomes increment `erpc_shadow_response_{identical,mismatch,error}_total`; mismatches log at error level with both hashes.

**Multi-tenancy on gRPC + query streams**: `x-erpc-project` metadata required, else `InvalidArgument` (erpc/grpc_server.go:138-141). `RequestProcessor.ProcessUnary` = GetProject → Validate → auth → GetNetwork → directive defaults → `project.Forward` (rate limit inside Forward) (erpc/request_processor.go:35-67); `ProcessQueryStream` additionally calls `project.AcquireRateLimitPermit` explicitly before the query executor since it bypasses `Forward` (erpc/request_processor.go:128-132).

**Admin RPCs touching projects** (erpc/admin.go): `erpc_project(params: [projectId])` → `{config, health}` where health = upstreams health + initializer status (erpc/admin.go:543-588, erpc/projects.go:83-92); `erpc_taxonomy` → all projects/networks/upstreams with aliases (erpc/admin.go:473-541); API-key CRUD resolves the project's database-auth connector via `consumerAuthRegistry.FindDatabaseConnector` (erpc/admin.go:82-100, auth/registry.go:99-111); cordon RPCs resolve upstreams through the project registry (erpc/admin.go:638-729). Admin auth itself is global (`admin.auth`), not per-project (erpc/erpc.go:83-89, erpc/admin.go:25-30); admin requests against an instance without `admin` config → `ErrAuthUnauthorized` "admin is not enabled for this project" (erpc/http_server.go:597-610).

**Healthcheck and projects**: no project in path → all projects evaluated (erpc/healthcheck.go:120-134); a project with zero initialized upstreams lazily prepares upstreams for its static networks before evaluating (erpc/healthcheck.go:160-181); providers-only projects with no initialized upstreams report healthy with an explanatory message (erpc/healthcheck.go:193-200).

**HTTP status mapping for project-related errors** (`determineResponseStatusCode`, erpc/http_server.go:1280-1317): 400 invalid path/unmarshal/invalid request; 401 `ErrAuthUnauthorized`; 404 `ErrProjectNotFound`/`ErrNetworkNotFound`; 429 `ErrAuthRateLimitRuleExceeded`/`ErrProjectRateLimitRuleExceeded`/`ErrNetworkRateLimitRuleExceeded`/capacity; everything else 200 (JSON-RPC app errors). The top-level fatal-error writer forces 200 for POST per JSON-RPC convention (erpc/http_server.go:770-773).

### Edge cases & gotchas

1. **Requests without an `Origin` header bypass CORS entirely** — by design; eRPC treats CORS as a soft deterrent, not a security boundary (erpc/http_server.go:1010-1018).
2. **Disallowed origins are NOT rejected**: non-OPTIONS requests proceed and get a normal JSON-RPC response, just without `Access-Control-*` headers; only the browser enforces blocking. OPTIONS gets a bare 204 (erpc/http_server.go:1035-1051; test erpc/http_server_test.go:3332-3376).
3. **`.` matches any single character in glob patterns** (go-wildcard v2 `matchByString`, `case '.'`): `allowedOrigins: ["https://erpc.cloud"]` also matches `https://erpcXcloud`. Same applies to aliasing `matchDomain` and all wildcard fields. There is no escape syntax.
4. **`forwardHeaders` stores values under the configured pattern, not the matched header name** (`nq.ForwardHeaders.Add(matchKey, value)`, erpc/http_server.go:487-491) — a wildcard pattern forwards upstream a literal header named like the pattern (e.g. `X-Custom-*`). Only exact-name patterns round-trip faithfully (clients/http_json_rpc_client.go:735-740).
5. **Project rate-limited requests never increment `erpc_network_request_received_total`** because `AcquireRateLimitPermit` runs before the counter (erpc/projects.go:113-125); use `erpc_rate_limits_total{origin="project"}` for those.
6. **OPERATOR FOOTGUN — `scoreMetricsWindowSize` stale source-code comments: comments say 10 minutes, actual fallback is 1 minute.** Two comments in the codebase directly contradict the live code value:
   - `common/config.go:530` states `"Defaults to 10m when zero"` in the GoDoc of the `ScoreMetricsWindowSize` field.
   - `erpc/projects_registry.go:131-132` states `"10m in production"` in the comment above the fallback assignment.
   - The **actual** package-level variable is `var ScoreMetricsWindowSize = 1 * time.Minute` (erpc/projects_registry.go:50).
   - An operator who reads the source or generated docs to understand the default will see "10m" and believe the health tracker window is 10 minutes. In reality it is 1 minute: the rolling window rotates every 6 seconds (window/10 buckets), upstream scoring reacts 10× faster than operators expect, and noise in low-traffic deployments will be much higher than the docs imply.
   - Practical consequence: if an operator deliberately omits `scoreMetricsWindowSize` to get the "stable 10m default" described in the comments, they are silently getting a 1m window. To actually get a 10-minute window they must set `scoreMetricsWindowSize: 10m` explicitly.
   - Trust the code (`erpc/projects_registry.go:50`), not the comments (`common/config.go:530`, `erpc/projects_registry.go:131-132`).
7. **All rate limiting fails open**: no store configured (`cache == nil` → allow, upstream/ratelimiter_budget.go:156-158), Redis still connecting (upstream/ratelimiter_registry.go:58-67), Redis call timeout (`limit_timeout`), or admission semaphore full (`admission_full`) (upstream/ratelimiter_budget.go:349-461). Deployments needing hard guarantees must monitor `erpc_rate_limiter_failopen_total`.
8. **`allowMethods` is not an allowlist by itself** — `shouldHandleMethod` starts `true`; without `ignoreMethods` everything is served regardless of `allowMethods` (erpc/http_server.go:499-529). To build an allowlist: `ignoreMethods: ["*"]` + `allowMethods: [...]` (same precedence documented for auth strategies, auth/authorizer.go:81-82).
9. **Unsupported-method responses normally carry `id: null`** due to the inverted `err != nil` check before reading `jrr.JSONRPC`/`jrr.ID` (erpc/http_server.go:533-537).
10. **`GetProject("")` returns `(nil, nil)`** — no error (erpc/projects_registry.go:112-115); HTTP guards earlier with "projectId is required in path or must be aliased" for non-admin (erpc/http_server.go:310-324). Code paths receiving a nil project must nil-check (`project != nil` guards throughout erpc/http_server.go:478,501,516,555).
11. **OPTIONS to a project without `cors` configured is not short-circuited** — the `r.Method == http.MethodOptions` early-return lives inside the `project.Config.CORS != nil` branch (erpc/http_server.go:343-347), so the OPTIONS request falls into JSON-RPC body parsing (and errors). Same for admin (erpc/http_server.go:295-301).
12. **CORS for a project can only run after project resolution succeeds**: unknown project → 404 with no CORS headers, which browsers may surface as a CORS failure rather than 404 (erpc/http_server.go:326-347).
13. **Any non-POST/non-OPTIONS request is a healthcheck**, e.g. `GET /myproject/evm/123` returns health JSON, not an RPC error (erpc/http_server.go:1001-1003; test "Root healthcheck GET" erpc/http_server_test.go:3424-3434).
14. **Lazy-loaded networks inherit `networkDefaults` and are appended into `project.Config.Networks`** at runtime (visible via admin `erpc_project`); verified including that only the defaulted fields (e.g. timeout) are set and others stay nil (TestProject_LazyLoadNetworkDefaults, erpc/projects_test.go:356-492; erpc/projects.go:58-77).
15. **First aliasing rule wins; rule-match errors are logged and skipped** (erpc/http_server.go:242-257). An explicit full 3-segment path can always override aliased preselection (erpc/http_server.go:893-897,924-928,954-958,976-980).
16. **Network aliases are per-project**; a duplicate alias targeting a different chain is ignored with a warning, keeping the first registration (erpc/networks_registry.go:334-348). Aliases must be unique within a project's static config (validation, common/validation.go:622-627).
17. **Zero-project configs still validate** because `SetDefaults` injects the `main` project before `Validate` runs (`LoadConfig` order: legacy translate → SetDefaults → Validate, common/config.go:111-129; common/validation.go:44-52).
18. **Project rate-limit test must align to the period window**: TestProject_Forward sleeps to the start of the next minute to avoid fixed-window rollover flakiness — a reminder the limiter is fixed-window, not sliding (erpc/projects_test.go:111-113).
19. **`projects[].rateLimitBudget` referencing an unknown budget fails startup** (`project.*.rateLimitBudget '<x>' does not exist in config.rateLimiters`, common/validation.go:640-644), but auth-strategy budgets and per-user (database/JWT-claim) budgets are resolved at request time and surface as runtime `ErrRateLimitBudgetNotFound` (upstream/ratelimiter_registry.go:215-225, auth/authorizer.go:129-132).
20. **CORS metrics label `project` actually contains the URL path**, so cardinality grows with distinct request paths and you cannot group by real project id from these counters alone (erpc/http_server.go:1020,1038,1069).
21. **Secret auth via `Authorization: Basic`** uses only the password half; the username is ignored (auth/http.go:40-53). The deprecated `?token=` query param still works and takes top precedence (auth/http.go:18-22).
22. **Auth rate-limit denial still returns the authenticated user** alongside the 429 error (`return user, err`, auth/registry.go:77-80), so `user` labels remain populated on rate-limit metrics.
23. **`exposedHeaders: nil` silently blocks browser access to all X-ERPC-* headers.** The call `strings.Join(nil, ", ")` returns `""`, so the server sends `Access-Control-Expose-Headers: ""`. Browsers interpret this as zero extra headers exposed. Any frontend JS call to `response.headers.get('X-ERPC-Cache')` returns `null`. To expose diagnostics to browser clients, enumerate the desired headers explicitly — for example `["X-ERPC-Cache", "X-ERPC-Upstream", "X-ERPC-Duration", "X-ERPC-Attempts"]`. There is no wildcard shorthand accepted by current browsers. (erpc/http_server.go:1057; no default in common/defaults.go:2824-2846).
24. **Legacy scoring fields are consumed and erased before `SetDefaults`** — after `TranslateFromConfig` runs, `LegacyProject` is nil on every `ProjectConfig`. Any code path that accesses `prj.LegacyProject` after `LoadConfig` (e.g. admin RPC, health endpoint) will always see nil; the field is effectively a load-time-only signal (common/legacy/translate.go:318-319, common/config.go:538 `yaml:"-" json:"-"`).
25. **Inert legacy fields (`scoreGranularity`, `scorePenaltyDecayRate`, `scoreMetricsMode`, `scoreRefreshInterval`) do NOT synthesize an eval.** A config that has ONLY these four fields (no `routingStrategy`, `scoreSwitchHysteresis`, `scoreMinSwitchInterval`, or network-level `evalFunction`/`resample*`) gets deprecation warnings logged but retains the full canonical default selection policy (`removeCordoned + keepHealthy + preferTag + sortByScore + stickyPrimary + probeExcluded`). Operators who expected the old minimal score-only behavior may be surprised by the more sophisticated default (common/legacy/translate.go:59-68, 155-164).
26. **`routingStrategy: "score-based"` (or any non-`"round-robin"` non-empty string) synthesizes a `sortByScore + stickyPrimary` eval.** The translator branch is `if strings.EqualFold(prj.RoutingStrategy, "round-robin")` → rotateBy; else → synthesizeScoreBasedEval. A per-network `selectionPolicy.eval` (new field) always wins; the legacy eval is not synthesized for networks that already have one (common/legacy/translate.go:80-83, 188-194).
27. **`scoreSwitchHysteresis` and `scoreMinSwitchInterval` are semantic fields that trigger eval synthesis even without `routingStrategy`.** If either is non-zero, `hasSemanticLegacy` returns true and a `sortByScore + stickyPrimary` eval is synthesized across all networks (common/legacy/translate.go:135-138). A config with only `scoreSwitchHysteresis: 0.2` (and no `routingStrategy`) silently synthesizes a full eval — this is intentional but can surprise operators who expected a scalar tuning knob.

- `erpc/projects.go` — `PreparedProject`: per-tenant facade; `Forward` (metrics, shadow fan-out), `AcquireRateLimitPermit` (project budget), `AuthenticateConsumer`, lazy network-config exposure, health info.
- `erpc/projects_registry.go` — `ProjectsRegistry`: builds/boots all projects; per-project tracker/policy-engine/auth/upstreams/networks wiring; `ScoreMetricsWindowSize` fallback var.
- `erpc/erpc.go` — `ERPC` root object: constructs shared registries (rate limiters, proxy pools, shared state) and the projects registry; admin auth registry.
- `erpc/http_server.go` — HTTP entry: aliasing, `parseUrlPath`, `handleCORS`, per-request pipeline (forwardHeaders, method gates, auth, forward), status-code mapping.
- `erpc/request_processor.go` — shared gRPC/query-stream pipeline (project resolution → auth → forward; explicit rate-limit permit for streams).
- `erpc/grpc_server.go` — gRPC project selection via `x-erpc-project`/`x-erpc-chain-id` metadata.
- `erpc/networks_registry.go` — per-project network lifecycle + alias registry; project-scoped cache binding.
- `erpc/admin.go` — `erpc_project`, `erpc_taxonomy`, API-key CRUD (per-project database auth connector), cordon RPCs.
- `erpc/healthcheck.go` — per-project/per-network health evaluation used by `/healthcheck` paths.
- `erpc/shadow.go` — project-layer shadow request execution/comparison.
- `erpc/block_heatmap.go` — per-project block-range heatmap metric emission from `Forward`.
- `common/config.go` — `ProjectConfig`, `CORSConfig`, `AuthConfig`/strategy configs, `AliasingConfig`, rate-limiter config types, `UserAgentTrackingMode`, legacy project fields.
- `common/defaults.go` — default `main` project injection, `ProjectConfig.SetDefaults` (provider fallbacks, upstream→provider conversion), `CORSConfig.SetDefaults`, `AdminConfig.SetDefaults` (admin CORS `*`).
- `common/validation.go` — `Config.Validate`/`ProjectConfig.Validate` (id, unique ids/aliases, budget existence), `AuthConfig.Validate`, `CORSConfig.Validate`.
- `common/matcher.go` — `WildcardMatch` boolean/glob grammar used by CORS, aliasing, method filters.
- `common/user.go` — `User{Id, RateLimitBudget}` attached by auth, drives `user` labels and per-user budgets.
- `common/errors.go` — `ErrProjectNotFound` (404), `ErrProjectAlreadyExists`, `ErrProjectRateLimitRuleExceeded` (429), `ErrAuthUnauthorized` (401), `ErrAuthRateLimitRuleExceeded` (429).
- `auth/registry.go`, `auth/authorizer.go`, `auth/http.go` — per-project auth strategy iteration, method filters, auth-level budgets, HTTP payload extraction.
- `upstream/ratelimiter_registry.go`, `upstream/ratelimiter_budget.go` — global budget store, Envoy-based fixed-window evaluation, fail-open machinery.
- `telemetry/metrics.go` — CORS counters, network request counters/histograms, unified rate-limit counter, auth/shadow metrics.
- Tests: `erpc/projects_test.go` (project rate limit, lazy networks, aliases), `erpc/http_server_test.go` (CORS allowed/disallowed/preflight, URL-path matrix).
- `common/legacy/translate.go` — `TranslateFromConfig` hook (LoadConfig pipeline, Phase 12): walks all projects, classifies legacy fields as semantic vs. inert, synthesizes `selectionPolicy.evalFunc`, clears stashes.
- `common/legacy/eval_synthesis.go` — `synthesizeScoreBasedEval`, `wrapLegacyEvalFunction`: JS source generators for score-based and round-robin legacy policies.
- `common/legacy/warnings.go` — per-field deprecation warning strings (anchored to https://docs.erpc.cloud/migration/selection-policy).
- `common/legacy/types.go` — `WidenedProject`, `WidenedUpstream`, `WidenedNetwork`: internal view types used by the translator (not exposed to runtime).
