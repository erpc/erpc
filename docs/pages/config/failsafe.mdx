# Failsafe

<LLMsTxtLink />

Failsafe policies handle intermittent upstream issues — timeouts, slowdowns, rate limits, transient errors, disagreement between upstreams. They live on **networks** (wraps the whole request lifecycle, including failover across upstreams) and on **upstreams** (one attempt against one endpoint).

**Five policies:**

- [`timeout`](#timeout-policy) — bound how long a request may take
- [`retry`](#retry-policy) — replay transient failures with backoff
- [`hedge`](#hedge-policy) — race a backup request when the primary is slow
- [`circuitBreaker`](#circuitbreaker-policy) — temporarily remove an upstream after sustained failure
- [`consensus`](/config/failsafe/consensus) — query multiple upstreams in parallel and require agreement

There's also a dedicated [integrity](/config/failsafe/integrity) doc for empty-response handling and data-correctness directives.

**You can scope each policy by:**

- **Method** — `matchMethod: "eth_call|trace_*"` (matcher syntax with `*`, `|`, `!`)
- **Finality** — `matchFinality: ["finalized", "unfinalized", "realtime", "unknown"]`. Note: there is **no `latest` value** — that's a block tag, not a finality state.

Policies in `failsafe[]` are evaluated in order; the first one whose `matchMethod` + `matchFinality` match the request is used.

## Minimum useful config

A reasonable production baseline: a network-wide timeout with retry across upstreams, and a hedge to short-circuit slow tails.

## Finality states (for `matchFinality`)

| State | What it means |
|---|---|
| `finalized` | Block past the chain's finalization horizon. Safe from reorgs. e.g. `eth_getBlockByNumber` on an old block, finalized `eth_getLogs` ranges. Relaxed failsafe is fine. |
| `unfinalized` | Recent block that could still reorg. Pending-block data also counts. e.g. `eth_getBlockByNumber("latest")` on a fresh block. May need more aggressive retries and shorter timeouts. |
| `realtime` | Data that updates every block: `eth_blockNumber`, `eth_gasPrice`, `eth_maxPriorityFeePerGas`, `net_peerCount`. Short timeouts + hedge are common. |
| `unknown` | Block number not derivable from request/response: `eth_getTransactionByHash`, `trace_transaction`, `debug_traceTransaction`. Data is typically immutable once mined; block context just isn't surfaced. |

<Callout type="warning">
	**`matchFinality: ["latest"]` is invalid.** `latest` is a block tag, not a finality state. Using it silently never matches. Use `realtime` or `unfinalized` instead.
</Callout>

## `timeout` policy

Two modes: fixed (`duration` only) or dynamic (quantile-based, with optional floor/ceiling). Dynamic timeouts use the observed latency distribution per upstream — the timeout adapts to the upstream's actual performance instead of being hard-coded.

<ConfigTabs
	path="projects > networks[] > failsafe[] > timeout"
	focus="9-13"
	yaml={`projects:
  - id: main
    networks:
      - architecture: evm
        evm:
          chainId: 1
        failsafe:
          # Dynamic p99 timeout with safety guard rails
          - matchMethod: "*"
            timeout:
              duration: 30s        # cold-start fallback
              quantile: 0.99       # timeout at p99 of observed latencies
              minDuration: 200ms   # never timeout faster than this
              maxDuration: 60s     # never wait longer than this`}
	ts={`import { createConfig } from "@erpc-cloud/config";

export default createConfig({
  projects: [{
    id: "main",
    networks: [{
      architecture: "evm",
      evm: { chainId: 1 },
      failsafe: [{
        matchMethod: "*",
        timeout: {
          duration: "30s",
          quantile: 0.99,
          minDuration: "200ms",
          maxDuration: "60s",
        },
      }],
    }],
  }],
});`}

## `hedge` policy

Speculative parallel request when the primary is slow. If the hedged copy beats the primary, the slower in-flight request is cancelled. Use a quantile to make the hedge delay adapt to observed latency.

<ConfigTabs
	path="projects > networks[] > failsafe[] > hedge"
	focus="7-11"
	yaml={`projects:
  - id: main
    networks:
      - architecture: evm
        evm: { chainId: 1 }
        failsafe:
          - matchMethod: "*"
            hedge:
              quantile: 0.9      # hedge at p90 of observed latency
              minDelay: 50ms     # never hedge faster than this
              maxCount: 2        # up to 2 parallel hedges`}
	ts={`import { createConfig } from "@erpc-cloud/config";

export default createConfig({
  projects: [{
    id: "main",
    networks: [{
      architecture: "evm",
      evm: { chainId: 1 },
      failsafe: [{
        matchMethod: "*",
        hedge: { quantile: 0.9, minDelay: "50ms", maxCount: 2 },
      }],
    }],
  }],
});`}
/>

**Worked example — bimodal latency with `quantile` + `minDelay` + `maxDelay`.**

Some chains have bimodal upstream latency: cached responses return in ~5ms while non-cached ones take 800ms–2s. A fixed `delay` either fires too early (constant cache-hit misses) or too late (slow upstreams never get hedged). Use quantile + guard rails instead:

```yaml
hedge:
  quantile: 0.95    # hedge at p95 of observed latency — adapts as upstreams warm/cool
  minDelay: 50ms    # never hedge before 50ms (avoids racing caches that return in <10ms)
  maxDelay: 2s      # never wait longer than 2s before launching a hedge
  maxCount: 2
```

The hedge fires at whichever is largest: the p95 observed latency, or `minDelay`. It never fires later than `maxDelay`. During cold start (no latency samples yet) the hedge falls back to `maxDelay` as the fixed delay.

## `circuitBreaker` policy

Temporarily removes an upstream from rotation after sustained failure. Three states: **closed** (normal), **open** (rejects immediately), **half-open** (test with limited traffic). Only valid at the **upstream** level — networks fail over across upstreams, not against themselves.

<ConfigTabs
	path="projects > upstreams[] > failsafe[] > circuitBreaker"
	focus="7-13"
	yaml={`projects:
  - id: main
    upstreams:
      - id: my-upstream
        endpoint: https://...
        failsafe:
          - matchMethod: "*"
            circuitBreaker:
              failureThresholdCount: 160       # open after 160 failures
              failureThresholdCapacity: 200    # ...out of a 200-sample window
              halfOpenAfter: 5m                # 5 min in open before testing again
              successThresholdCount: 3         # close again after 3 successes
              successThresholdCapacity: 10     # ...out of a 10-sample window`}
	ts={`import { createConfig } from "@erpc-cloud/config";

export default createConfig({
  projects: [{
    id: "main",
    upstreams: [{
      id: "my-upstream",
      endpoint: "https://...",
      failsafe: [{
        matchMethod: "*",
        circuitBreaker: {
          failureThresholdCount: 160,
          failureThresholdCapacity: 200,
          halfOpenAfter: "5m",
          successThresholdCount: 3,
          successThresholdCapacity: 10,
        },
      }],
    }],
  }],
});`}

## Disabling a policy

Set the policy's value to `null` (YAML) or `undefined`/omit (TypeScript) to opt out of a default that would otherwise apply.

```yaml
failsafe:
  - matchMethod: "*"
    timeout:
      duration: 30s
    retry: null            # explicitly disable retry on this method
```

### `FailsafeConfig` — top-level fields

| Field | Type | Notes |
|---|---|---|
| `matchMethod` | string | Matcher pattern. Defaults to `"*"`. Supports `*` (wildcard), `\|` (OR), `!` (NOT). |
| `matchFinality` | `("finalized"\|"unfinalized"\|"realtime"\|"unknown")[]` | When omitted, matches every finality. **Do not use `"latest"` here** — that's a block tag, not a state. |
| `retry` | RetryPolicyConfig | See "Retry policy" below. |
| `circuitBreaker` | CircuitBreakerPolicyConfig | Upstream-only. See "Circuit breaker policy" below. |
| `timeout` | TimeoutPolicyConfig | See "Timeout policy" below. |
| `hedge` | HedgePolicyConfig | See "Hedge policy" below. |
| `consensus` | ConsensusPolicyConfig | See [Consensus](/config/failsafe/consensus). |

Each policy is independent — you can set any subset on a single `failsafe[]` entry. Evaluation order within `failsafe[]` is top-to-bottom; first match wins.

### Where each policy is valid

| Policy | Network level | Upstream level | Notes |
|---|---|---|---|
| `timeout` | ✅ | ✅ | Network timeout covers the **full** lifecycle (including every upstream retry). Upstream timeout bounds one attempt. |
| `retry` | ✅ | ✅ | Network-level retries rotate **across** upstreams. Upstream-level retries hit the **same** upstream. Empty-result retries (`emptyResultAccept`, etc.) only fire at the network level. |
| `hedge` | ✅ | ✅ | Common at network level (race across upstreams). |
| `circuitBreaker` | ❌ | ✅ | Trips one upstream out of the rotation; the network's selection policy then routes elsewhere. |
| `consensus` | ✅ | ✅ | Most commonly network-level; per-upstream usage is rare. |

### Retryable vs non-retryable errors (canonical list)

**Retryable** (`retry` will replay these):

- HTTP `5xx` from the upstream
- HTTP `408` (request timeout)
- HTTP `429` (rate limit) — but prefer `rateLimitAutoTune` for sustained pressure
- Network errors (TCP reset, DNS failure)
- Empty responses for methods NOT in `retry.emptyResultAccept`, when `retryEmpty` directive is set
- Block-unavailable conditions where the request's block reference is beyond every upstream's known head

**Non-retryable** (single-attempt; never retried):

- HTTP `4xx` other than `408`/`429`
- `MethodNotSupported` from the upstream
- Empty responses for methods in `retry.emptyResultAccept` at-or-below `emptyResultConfidence` horizon
- Write methods (`eth_sendRawTransaction`, `eth_sendTransaction`) — unless `evm.idempotentTransactionBroadcast` is enabled on the network

### `RetryPolicyConfig` — every field

| Field | Default | Notes |
|---|---|---|
| `maxAttempts` | `3` | Total attempts including the first. `maxAttempts: 1` disables retry. |
| `delay` | `0ms` | Initial wait before the second attempt. |
| `backoffFactor` | `1.5` | Multiplier between successive delays. `1.0` = constant delay. |
| `backoffMaxDelay` | `5s` | Ceiling on the computed delay. |
| `jitter` | `0ms` | Random `[0, jitter)` added on top of each delay. Prevents thundering herd. |
| `emptyResultAccept` | `["eth_getLogs", "eth_call"]` (effective default) | Methods where empty IS valid data — won't be retried as if it were missing. |
| `emptyResultConfidence` | `blockHead` | `finalizedBlock` or `blockHead`. Decides whether an empty response to a not-yet-final block is retried. `finalizedBlock` is more conservative (only finalized empties are trusted). |
| `emptyResultMaxAttempts` | = `maxAttempts` | Separate cap for empty-result retries. |
| `emptyResultDelay` | = `delay` | Delay specifically between empty-result retries. |
| `blockUnavailableDelay` | dynamic (network block-time × `blockUnavailableDelayMultiplier`) | Wait when the requested block hasn't propagated to any upstream yet. |
| `emptyResultIgnore` | — | **Deprecated** alias for `emptyResultAccept`. Still accepted with a deprecation warning. |

See [`retry` walkthrough](/preview-retry) for empty-result decision flow, three real-world examples, and metrics to watch.

### `TimeoutPolicyConfig` — every field

| Field | Default | Notes |
|---|---|---|
| `duration` | none | Fixed timeout. Used as cold-start fallback when quantile-based mode is enabled. |
| `quantile` | none | Latency percentile (`0 < q < 1`). When set, the timeout adapts to the upstream's observed latency at this percentile. `0.99` is typical. |
| `minDuration` | none | Floor for the computed timeout. Prevents false timeouts when observed latencies are very low. |
| `maxDuration` | none | Ceiling. May serve as the cold-start fallback if `duration` is omitted. |

**Recommendation:** for most production setups, `duration: 30s, quantile: 0.99` is enough. Add `minDuration` if your upstream is sometimes blazingly fast and you want to avoid a sub-100 ms hair-trigger.

Metrics: `erpc_network_timeout_duration_seconds` (histogram of dynamically computed timeouts per method).

### `HedgePolicyConfig` — every field

| Field | Default | Notes |
|---|---|---|
| `delay` | none | Fixed delay before issuing a hedge. |
| `maxCount` | `1` | Max number of hedged requests (in addition to the primary). |
| `quantile` | none | Latency percentile; when set, the hedge delay is the percentile of observed latencies. |
| `minDelay` | none | Floor for the hedge delay. |
| `maxDelay` | none | Ceiling. |

Combine `quantile` + `minDelay` to make the hedge delay adapt to upstream performance while staying above a minimum (avoid hedging when even the slow case is fast).

### `CircuitBreakerPolicyConfig` — every field

| Field | Notes |
|---|---|
| `failureThresholdCount` | Open the breaker after this many failures... |
| `failureThresholdCapacity` | ...within this rolling sample window. e.g. 160/200 = "trip after 80% failure across the last 200 requests". |
| `halfOpenAfter` | Duration in open state before transitioning to half-open. |
| `successThresholdCount` | Close the breaker again after this many successes... |
| `successThresholdCapacity` | ...within this success-sample window. e.g. 3/10 = "close after 3 successful probes in 10 tries". |

Only valid at the **upstream** level. Networks don't trip — they rotate across upstreams instead.

### `ConsensusPolicyConfig` summary

See the dedicated [consensus page](/config/failsafe/consensus) for the full reference. High-level fields:

| Field | Notes |
|---|---|
| `maxParticipants` | Number of upstreams to query in parallel. |
| `agreementThreshold` | Minimum upstreams that must agree. Default = `maxParticipants/2 + 1`. |
| `disputeBehavior` | What to do on disagreement: `returnError`, `acceptMostCommonValidResult`, `preferBlockHeadLeader`, `onlyBlockHeadLeader`. |
| `lowParticipantsBehavior` | What to do when fewer than `agreementThreshold` upstreams reply. |
| `punishMisbehavior` | `{ disputeThreshold, disputeWindow, sitOutPenalty }` — temporal penalty for disputing upstreams. |
| `disputeLogLevel` | `trace`/`debug`/`info`/`warn`/`error`. Verbosity of dispute logs. |
| `ignoreFields` | Map of method → list of dot-path fields to exclude from comparison (e.g. `blockTimestamp` differs across upstreams). |
| `preferNonEmpty` | Bias the dispute resolution toward non-empty results. |
| `preferLargerResponses` | Bias toward larger response bodies. |
| `preferHighestValueFor` | Map of method → field-path list. Picks the highest numeric value; used for nonces. |
| `misbehaviorsDestination` | File or S3 export of disputing requests for offline review. |
| `fireAndForget` | Don't cancel in-flight requests when consensus short-circuits (useful for broadcasts). |

### Per-method scoping recipes

**Different policy per finality**:

```yaml
failsafe:
  - matchMethod: "*"
    matchFinality: ["realtime", "unfinalized"]
    timeout: { duration: 5s }
    retry:   { maxAttempts: 3, delay: 100ms }
  - matchMethod: "*"
    matchFinality: ["finalized"]
    timeout: { duration: 60s }     # tolerate long backfill reads
    retry:   { maxAttempts: 5, delay: 200ms }
  - matchMethod: "*"
    matchFinality: ["unknown"]      # tx-hash keyed (receipts, traces by hash)
    timeout: { duration: 30s }
    retry:   { maxAttempts: 3 }
```

**Different policy per method group**:

```yaml
failsafe:
  - matchMethod: "trace_*|debug_*"   # expensive — don't multiply
    timeout: { duration: 60s }
    retry:   { maxAttempts: 1 }
  - matchMethod: "eth_getLogs"
    timeout: { duration: 30s }
    retry:   { maxAttempts: 3, delay: 100ms }
  - matchMethod: "*"
    timeout: { duration: 15s }
    retry:   { maxAttempts: 3 }
```

### Real-world example — indexer chasing tip with broad empty retries

Network-wide retry-empty (caches will catch up shortly), tight per-method scoping for the long-tail backfill.

```yaml
projects:
  - id: indexer
    networks:
      - architecture: evm
        evm: { chainId: 1 }
        directiveDefaults:
          retryEmpty: true        # treat empty as retryable across the board
        failsafe:
          - matchMethod: "eth_getLogs|eth_call"
            retry:
              maxAttempts: 5
              delay: 100ms
              backoffFactor: 1.2
              jitter: 50ms
              emptyResultConfidence: finalizedBlock
              emptyResultMaxAttempts: 2
          - matchMethod: "*"
            retry: { maxAttempts: 3, delay: 200ms }
            timeout: { duration: 30s }
```

### Common pitfalls

- **`matchFinality: ["latest"]`** — silently never matches. Valid values are `finalized`, `unfinalized`, `realtime`, `unknown`.
- **Mixing upstream + network retry without thinking about the product** — `upstream.retry.maxAttempts: 3` × `network.retry.maxAttempts: 3` = up to 9 attempts per request. Easy to accidentally 9× your upstream traffic.
- **Network timeout shorter than `upstream.timeout × maxAttempts`** — the network gives up before the upstream's retry budget is exhausted. Set the network timeout generously.
- **CircuitBreaker at network level** — only valid at upstream level; the parser will accept it but it has no effect at network level.
- **`emptyResultIgnore` is deprecated** — rename to `emptyResultAccept` in existing configs; old configs still work with a deprecation warning. For network-wide empty-retry control, use `directiveDefaults.retryEmpty: false` (or per-request `?retryEmpty=false`) instead of relying on `retry.emptyResultIgnore`.
- **Single-object legacy `failsafe: { ... }` form** — still accepted, but the array form with `matchMethod: "*"` is canonical. The single-object form is implicitly `matchMethod: "*"`.
- **`retry.delay: 0ms` doesn't disable retry** — it means "no wait between attempts". Use `maxAttempts: 1` to disable retry entirely.
- **Write methods aren't retried** even when retry is configured. Set `network.evm.idempotentTransactionBroadcast: true` if you want `eth_sendRawTransaction` to be safe under retry/hedge.
