# Selection policy

The **Selection Policy** is the single mechanism that decides which upstreams
serve which methods, in what order. A network-level JavaScript function
(`eval`) returns an ordered list of upstreams; that list IS the routing
decision — the request loop tries them in order, missing means excluded.

<a id="priority--routing"></a>

<Callout type="info">
This replaces the dual "scoring mechanism" + "selection policy" pair that
shipped in earlier releases. If you are upgrading, see the
[migration guide](/migration/selection-policy) — legacy YAML continues to
load via the translator for two more minor releases.
</Callout>

## How it works

```
                              every `evalInterval`
                                       ▼
upstreams + ctx (cross-tick state) ──► eval() ──► ordered upstream[]
                                                       │
                                                       ▼
                            Network.Forward consumes the ordered list
                            on every request (wait-free atomic load)
```

- The eval runs on a per-(network, method) timer.
- The output is cached atomically; the request path reads it in O(1).
- Each tick produces a **decision record** retained for `decisionHistory`,
  joined to requests via `decision_id` so you can answer
  *"why was upstream X used at time T for method M?"* from the
  `/admin/selection/explain/<requestId>` endpoint.

projects:

### Fields

| Field | Type | Description |
|---|---|---|
| `evalInterval` | `Duration` | How often the eval runs. Default `1s`. Setting to `0` disables the ticker (manual reevaluation only — useful for tests). |
| `evalTimeout` | `Duration` | Wall-clock cap on each eval. Must be `< evalInterval`. Default `100ms`. On timeout, the prior cache is retained and the failure is logged. |
| `evalPerMethod` | `bool` | If `true`, a separate eval + cache is maintained per method. Default `false`. |
| `decisionHistory` | `Duration` | Rolling decision-record retention. Default `5m`. The ring buffer is sized as `decisionHistory / evalInterval`. |
| `eval` | `string` | JavaScript function body. Must `return` an `Upstream[]`. If omitted, the [default policy](#default-policy) applies. |

`selectionPolicy` is **network-level**. There are no project-level routing
settings.

## Default policy

When `selectionPolicy.eval` is omitted, eRPC applies this embedded default:

```js
(upstreams, ctx) =>
  upstreams
    .sortByScore(BALANCED)
    .preferGroup('default', { minHealthy: 1, fallback: 'fallback' })
    .stickyPrimary({ hysteresis: 0.10, minSwitchInterval: '30s' })
    .probeExcluded({ reAdmitAfter: '5m', maxConcurrent: 1 })
```

Behavior:

1. **`sortByScore(BALANCED)`** — order by penalty (lower = better). Weights
   are `errorRate=8, respLatency=4, throttledRate=3, blockHeadLag=2,
   finalizationLag=1, misbehaviors=6`. See the [stdlib reference](/config/projects/selection-policy-stdlib#sortbyscore)
   for the full preset table.
2. **`preferGroup('default', { minHealthy: 1, fallback: 'fallback' })`** —
   keep only upstreams in group `default` if at least one is healthy. If
   `default` is empty, fall back to upstreams in group `fallback`.
3. **`stickyPrimary({ hysteresis: 0.10, minSwitchInterval: '30s' })`** —
   keep the current primary unless a challenger's score is at least 10%
   lower AND at least 30s have passed since the last switch.
4. **`probeExcluded({ reAdmitAfter: '5m', maxConcurrent: 1 })`** — every
   tick, deterministically re-admit one upstream that has been excluded
   for 5+ minutes so its metrics refresh.

The default policy source is embedded in the binary and served at
`GET /admin/selection/default-policy` for verification.

## Eval inputs

Inside the `eval` body, two variables are in scope:

### `upstreams: Upstream[]`

```ts
type Upstream = {
  readonly id: string
  readonly vendor: string            // e.g. "alchemy", "infura", "drpc"
  readonly type: 'evm' | string      // upstream architecture
  readonly endpoint: string          // redacted

  readonly config: UpstreamConfig    // raw user config (group, etc.)
  readonly metrics: UpstreamMetrics  // snapshot from tick start

  // Attached by std-lib steps; readable by later steps and decision records.
  readonly score?: number            // set by sortByScore (lower = better)
  readonly penaltyBreakdown?: ScoreBreakdown
  readonly annotations?: string[]    // set by .annotate() / .mark() / std-lib
}

type UpstreamMetrics = {
  errorRate: number                  // 0..1
  errorsTotal: number
  requestsTotal: number
  throttledRate: number              // 0..1
  misbehaviorRate: number
  p50ResponseSeconds: number
  p70ResponseSeconds: number
  p90ResponseSeconds: number
  p95ResponseSeconds: number
  p99ResponseSeconds: number
  blockHeadLag: number               // block-number delta
  finalizationLag: number            // block-number delta
  cordonedReason: string | null
}
```

The `metrics` object is a **snapshot** captured once at the tick start, so
all chained std-lib steps see a consistent view.

### `ctx: EvalContext`

```ts
type EvalContext = {
  network: string                    // e.g. "evm:1"
  method: string                     // "*" if evalPerMethod=false
  finality: 'realtime' | 'unfinalized' | 'finalized' | 'unknown'
  now: number                        // unix ms

  // Cross-tick state, fed back from the previous tick's output:
  previousOrder: string[]            // upstream IDs from last tick
  previousExcluded: string[]         // upstream IDs absent from last tick
  lastSwitchAt: number | null        // unix ms of last sticky-primary switch
  excludedSince: { [id: string]: number }
  tickCount: number                  // monotonic, resets on slot reset
}
```

`ctx` is the **only** carrier of cross-tick state. The engine itself stores
nothing beyond what is serialized in/out of `ctx`, making evals testable as
pure functions.

### Globals

Beyond `upstreams` and `ctx`:

- **Constants:** `BALANCED`, `PREFER_FASTER`, `PREFER_FEWER_ERRORS`,
  `PREFER_FRESHER_HEAD`, `PREFER_LESS_THROTTLED`, `PREFER_CHEAP`; finality
  states `REALTIME` / `UNFINALIZED` / `FINALIZED` / `UNKNOWN`; reason
  codes `REASON_LAG`, `REASON_ERROR_RATE`, etc.
- **`process.env`** — environment variables, read-only.
- **`console`** — `log` / `info` / `warn` / `error` → debug-level structured log.
- Standard ECMAScript: `Math`, `Date`, `JSON`, `Number`, `String`, `Array`, `Object`.

## Standard library

Every method on `upstreams` (and chain results) returns an `Upstream[]`
so they compose. Full catalogue at the
[stdlib reference](/config/projects/selection-policy-stdlib); here is the
at-a-glance map by category:

| Category | Methods |
|---|---|
| **Identity** | `.where`, `.byId`, `.byGroup`, `.byVendor`, `.byType`, `.exclude*` |
| **Health filters** | `.removeByErrorRate`, `.removeByLag`, `.removeByLatency`, `.removeByThrottling`, `.removeByMisbehavior`, `.removeCordoned`, `.removeByMinRequests`, `.keepHealthy` |
| **Generic functional** | `.filter`, `.reject`, `.partition`, `.unique`, `.union`, `.intersect`, `.difference`, `.isEmpty` |
| **Sorting** | `.sortByScore` (preset / weights map / `(u) => weights`), `.sortBy`, `.sortByLatency`, `.sortByErrorRate`, `.sortByThrottling`, `.sortByHeadLag`, `.sortByFinalizationLag`, `.sortByMisbehavior` |
| **Randomization** | `.shuffle`, `.rotateBy` |
| **Stability (cross-tick)** | `.stickyPrimary` |
| **Grouping** | `.preferGroup`, `.preferVendor` |
| **Slicing** | `.pickTop`, `.pickBottom`, `.dropTop`, `.dropBottom`, `.take`, `.skip` |
| **Probing** | `.probeExcluded`, `.forceInclude` |
| **Cooldown** | `.cooldown` |
| **Combinators** | `.if`, `.unless`, `.whenEmpty`, `.whenNotEmpty`, `.fallbackTo`, `.ensureMin` |
| **Debug** | `.tap`, `.annotate`, `.label`, `.mark`, `.dump` |

## Common patterns

### Cost-tier routing

```js
// Cheap upstreams first; fall back to fast tier only if cheap pool is unhealthy.
(upstreams, ctx) =>
  upstreams
    .removeByErrorRate(0.5)
    .preferGroup('cheap', { minHealthy: 2, fallback: 'fast' })
    .sortByScore(BALANCED)
```

### Method-specific routing

```js
(upstreams, ctx) => {
  if (methodMatches(['eth_getLogs', 'eth_getBlockByNumber'])) {
    return upstreams.byGroup('archive').sortByScore(PREFER_FASTER);
  }
  return upstreams.sortByScore(BALANCED).stickyPrimary();
}
```

### Finality-aware weighting

```js
(upstreams, ctx) =>
  upstreams.sortByScore(
    ctx.finality === FINALIZED ? PREFER_FASTER : PREFER_FRESHER_HEAD
  )
```

### Time-of-day rotation

```js
// Drain expensive vendors during business hours, lean on them at night.
(upstreams, ctx) =>
  upstreams.preferGroup(
    inWindow('09:00', '18:00') ? 'cheap' : 'fast',
    { minHealthy: 1, fallback: 'cheap' },
  ).sortByScore(BALANCED)
```

More examples in the [stdlib reference](/config/projects/selection-policy-stdlib#examples).

## Observability

| Surface | What it gives you |
|---|---|
| `GET /admin/selection/<net>/<method>` | Latest decision record (input metrics, eval duration, output order, exclusion reasons, diff vs prior tick) |
| `GET /admin/selection/<net>/<method>?since=10m` | Sparse log of decisions in the window |
| `GET /admin/selection/explain/<requestId>` | Joins a specific request to the decision active at its time — answers *"why was upstream X used for this request?"* |
| `GET /admin/selection/default-policy` | Source of the embedded default policy |
| `POST /admin/selection/<net>/<method>/reeval` | Force a re-evaluation |
| `POST /admin/selection/<net>/<method>/reset` | Drop cross-tick state, re-evaluate immediately |
| `erpc_selection_position{project, network, method, upstream}` | Gauge: 0 = primary, 1+ = runners-up, -1 = excluded |
| `erpc_selection_rejection_total{step, ...}` | Counter: per-std-lib-step rejections |
| `erpc_selection_primary_switch_total{from, to, ...}` | Counter: primary upstream changes |
| `erpc_selection_eval_duration_seconds{...}` | Histogram: per-tick eval latency |
| `erpc_selection_eligible_upstreams{...}` | Gauge: count returned by the most recent tick |

Every request log contains the `decision_id` of the decision active when it
was served. Combine with the `/explain/<requestId>` endpoint for full
attribution.
