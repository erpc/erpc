# Selection policy

A single JavaScript function (`eval`) per network returns the **ordered**
list of upstreams that should serve requests. The order IS the routing
decision: position 0 is the primary, position N is the Nth retry,
anything missing is excluded.

<a id="priority--routing"></a>

<Callout type="info">

## How it works

```
                              every `evalInterval`
                                       ▼
upstreams + ctx (cross-tick state) ──► eval() ──► ordered upstream[]
                                                       │
                                                       ▼
                            Network.Forward consumes the list on every
                            request (wait-free atomic load, O(1))
```

- The eval runs on a per-(network, method) timer.
- Output is cached atomically; the request path reads it lock-free.
- Every tick produces a **decision record** joined to requests via
  `decision_id` — `GET /admin/selection/explain/<requestId>` answers
  *"why was upstream X used at time T for method M?"*.

| Field | Type | Default | Notes |
|---|---|---|---|
| `evalInterval` | Duration | `1s` | `0` disables the ticker (test only). |
| `evalTimeout` | Duration | `100ms` | Must be `< evalInterval`. Prior cache retained on timeout. |
| `evalPerMethod` | bool | `false` | Separate eval + cache per method. |
| `evalFunc` | string | (built-in default) | JS function returning `Upstream[]`. Signature `(upstreams, ctx) => Upstream[]`. |

> **Investigations.** There is no in-memory decision-history buffer
> or admin endpoint for inspecting past evals. Per-tick reasoning is
> exposed via OTLP tracing spans and the `policy_selection_*`
> Prometheus metric family (selection counters, primary-switch counters,
> rejection counters, eval-duration histogram). Use your existing
> tracing / metrics stack for incident investigation.

## The default policy

Omitting `evalFunc` applies a production-hardened, chain-agnostic policy.
Each step addresses a distinct failure class — the *absence* of any
one of them lets a degraded upstream keep receiving traffic:

```js
(upstreams, ctx) =>
  upstreams
    .removeCordoned()                  // 1
    .keepHealthy({                     // 2 — composite filter
      maxErrorRate:     0.5,
      maxBlockHeadLag:  16,
      maxP95Ms:         10_000,
      maxThrottledRate: 0.3,
    })
    .whenEmpty(() => upstreams)        // 3 — safety net
    .preferTag('!tier:fallback', {     // 4 — tier
      minHealthy: 1,
      fallback:   'tier:fallback',
    })
    .sortByScore(BALANCED)             // 5 — rank survivors
    .if(ctx.finality === REALTIME,     // 6 — sticky for live-latest only
        u => u.stickyPrimary({ hysteresis: 0.10, minSwitchInterval: '30s' }),
        u => u)
    .probeExcluded({                   // 7 — fast re-admit
      reAdmitAfter:  '90s',
      maxConcurrent: 2,
    })
```

Step-by-step:

1. **Drop cordoned** — failsafe / circuit-breaker has marked the upstream bad.

2. **Keep healthy** — a single composite filter that drops a degraded
   upstream BEFORE it can win on score:
   - `maxErrorRate: 0.5` — anything erroring more than half the time is broken on any chain.
   - `maxBlockHeadLag: 16` — ~1 min behind tip on Ethereum mainnet, ~32 s on a 2 s chain. Clearly degraded for `latest`-style queries.
   - `maxP95Ms: 10_000` — catches a slow vendor before its blended percentile poisons the pool's hedge / consensus timeouts.
   - `maxThrottledRate: 0.3` — the vendor is rate-limiting us; reroute before quota burns.

   Filtering on these signals (rather than only ranking) is the key
   defence against "an upstream is slow / erroring but its score is
   still in the top tier".

3. **Safety net** — if step 2 wiped everything (project-wide outage), serve from the raw set rather than fail closed.

4. **Tier** — `group !== 'fallback'` is the primary tier; `group === 'fallback'` is the second tier. One config line per fallback upstream.

5. **Score** — `BALANCED` weights (`errorRate=8, respLatency=4, throttledRate=3, blockHeadLag=2, finalizationLag=1, misbehaviors=6`). Ranks the survivors of step 2.

6. **Sticky primary for REALTIME only** — keeping the current primary across ticks prevents flapping for live-latest queries. Finalized/unfinalized requests are reorg-tolerant; pinning to a degrading primary on those just amplifies incident impact.

7. **Probe excluded** — re-admit two excluded upstreams every 90 s so they get a chance to prove they've recovered. Typical transient-blip windows (rate-limit reset, regional networking dip) recover in 60-90 s.

Source served at `GET /admin/selection/default-policy`.

### Loosening the default for low-traffic / dev setups

If the defaults above are too strict for your environment — e.g. you
have only one upstream per network and you'd rather a degraded
upstream serve traffic than fail closed — write an explicit `eval`
that loosens or replaces specific steps. Common loosener:

```js
(upstreams, ctx) =>
  upstreams
    .removeCordoned()
    .removeByErrorRate(0.8)            // looser error threshold
    .whenEmpty(() => upstreams)
    .sortByScore(BALANCED)
    .stickyPrimary({ hysteresis: 0.10, minSwitchInterval: '30s' })
    .probeExcluded({ reAdmitAfter: '5m', maxConcurrent: 1 })
```

This was the default in earlier eRPC versions; ship it as your
`eval` if you depended on that behavior.

## Eval inputs

Inside the function body, two variables are in scope: `upstreams` and `ctx`.

```ts
type Upstream = {
  readonly id: string
  readonly vendor: string                  // "alchemy", "infura", "drpc"
  readonly type: 'evm' | string
  readonly endpoint: string                // redacted

  readonly config: UpstreamConfig          // user config (group, etc.)
  readonly metrics: UpstreamMetrics        // tick-start snapshot

  // Attached by std-lib steps:
  readonly score?: number                  // sortByScore (lower = better)
  readonly penaltyBreakdown?: ScoreBreakdown
  readonly annotations?: string[]
}

type UpstreamMetrics = {
  errorRate: number                        // 0..1
  errorsTotal: number
  requestsTotal: number
  throttledRate: number                    // 0..1
  misbehaviorRate: number
  p50ResponseSeconds: number
  p70ResponseSeconds: number
  p90ResponseSeconds: number
  p95ResponseSeconds: number
  p99ResponseSeconds: number
  blockHeadLag: number                     // BLOCK-number delta, NOT seconds
  finalizationLag: number                  // block-number delta
  cordonedReason: string | null
}

type EvalContext = {
  network: string                          // "evm:1"
  method: string                           // "*" unless evalPerMethod=true
  finality: 'realtime' | 'unfinalized' | 'finalized' | 'unknown'
  now: number                              // unix ms

  // Cross-tick state — the ONLY carrier of state between ticks.
  previousOrder: string[]
  previousExcluded: string[]
  lastSwitchAt: number | null
  excludedSince: { [id: string]: number }
  tickCount: number
}
```

The `metrics` object is captured ONCE at tick start, so chained std-lib
steps see a consistent view.

Globals: `BALANCED`, `PREFER_FASTER`, `PREFER_FEWER_ERRORS`,
`PREFER_FRESHER_HEAD`, `PREFER_LESS_THROTTLED`, `PREFER_CHEAP`;
`REALTIME`/`UNFINALIZED`/`FINALIZED`/`UNKNOWN`;
`REASON_*` reason codes; `process.env`; `console.log/info/warn/error`;
standard ECMAScript.

## Common patterns

### Cost-tier routing
```js
// Cheap pool first; fall through to fast pool when cheap is exhausted.
// Upstreams declare their tier via `tags: [tier:cheap]` or `tags: [tier:fast]`.
(upstreams, ctx) =>
  upstreams
    .removeByErrorRate(0.5)
    .preferTag('tier:cheap', { minHealthy: 2, fallback: 'tier:fast' })
    .sortByScore(BALANCED)
```

### Per-method routing
```js
// Upstreams declare their capability via `tags: [tier:archive]`.
(upstreams, ctx) => {
  if (methodMatches(['eth_getLogs', 'eth_getBlockByNumber']))
    return upstreams.byTag('tier:archive').sortByScore(PREFER_FASTER)
  return upstreams.sortByScore(BALANCED).stickyPrimary()
}
```

### Per-upstream weights (legacy `scoreMultipliers` replacement)
```js
const w = {
  alchemy: { errorRate: 8, respLatency: 12 },
  drpc:    { errorRate: 8, respLatency: 4 },
}
(upstreams, ctx) =>
  upstreams.sortByScore((u) => w[u.vendor] || BALANCED)
```

More patterns + every method explained at
[selection policy stdlib](/config/projects/selection-policy-stdlib#examples).

## Observability

| Surface | What it gives you |
|---|---|
| `GET /admin/selection/<net>/<method>` | Latest decision (input metrics, eval duration, output order, exclusions, diff vs prior tick) |
| `GET /admin/selection/<net>/<method>?since=10m` | Sparse log of recent decisions |
| `GET /admin/selection/explain/<requestId>` | Join a request to the decision active at its time |
| `GET /admin/selection/default-policy` | Source of the built-in default |
| `POST /admin/selection/<net>/<method>/reeval` | Force a tick |
| `erpc_selection_position{...}` (gauge) | 0 = primary, 1+ = runner-up, -1 = excluded this tick |
| `erpc_selection_rejection_total{step, ...}` | Per-std-lib-step rejections |
| `erpc_selection_primary_switch_total{from, to, ...}` | Primary changes |
| `erpc_selection_eval_duration_seconds{...}` | Per-tick eval latency |
| `erpc_selection_eligible_upstreams{...}` | Count returned by the most recent tick |

Every request log includes the `decision_id` of the decision active when
it was served — pair with `/explain/<requestId>` for full attribution.
