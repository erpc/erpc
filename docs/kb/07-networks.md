## L1 — Capability (CTO view)

A **network** is eRPC's per-chain routing domain inside a project: one `Network` object per `(architecture, chainId)` pair (currently only `evm`), identified by a canonical `networkId` string like `evm:42161`. Networks own the full request pipeline — static responses, request multiplexing (in-flight dedup), cache lookup, upstream ordering via the selection-policy engine, failsafe (retry/hedge/timeout/consensus) execution, block-availability gating, and served-tip ("latest"/"finalized") computation. Operators get three ergonomic wins: (1) **zero-config chains** — networks are created lazily on first request and upstreams are auto-generated from providers for any chain the providers support; (2) **human-friendly URLs** — a network `alias` ("arbitrum") routes in place of `evm/42161`, and domain-based aliasing rules can pre-select project/architecture/chain entirely; (3) **`networkDefaults`** — project-wide defaults inherited by every network (static or lazy) so per-chain blocks stay minimal.

## L2 — Mechanics (staff-engineer view)

**Architecture concept.** `NetworkArchitecture` is a string enum with a single valid value, `evm` (`common/network.go:L12-16`); `IsValidArchitecture` accepts only `"evm"` (`common/network.go:L35-37`). The `common.Network` interface (`common/network.go:L18-33`) exposes `Id()`, `Label()`, `Architecture()`, `Forward()`, `GetFinality()`, plus EVM-specific head accessors (`EvmHighestLatestBlockNumber`, `EvmHighestFinalizedBlockNumber`, `EvmLeaderUpstream`) that the interface comment flags as candidates for a future `EvmNetwork` split. The concrete implementation is `erpc.Network` (`erpc/networks.go:L31-78`). The canonical id format is `evm:<chainId>` produced by `util.EvmNetworkId` (`util/ids.go:L11-13`); `util.IsValidNetworkId` accepts only an `evm:`-prefixed integer (`util/ids.go:L21-27`). Architecture-specific behavior is dispatched by `switch` on `cfg.Architecture` at every seam: `prepareNetwork` wires the EVM cache DAL and rejects unknown architectures (`erpc/networks_registry.go:L314-321`), `prepareRequest` normalizes EVM JSON-RPC and errors on anything else (`erpc/networks.go:L1605-1630`), `doForward` runs EVM pre/post-forward hooks (`erpc/networks.go:L1745-1756`), and `PreparedProject.doForward` runs project-level EVM pre-forward (`erpc/projects.go:L256-268`).

**Object hierarchy & construction.** `ProjectsRegistry.RegisterProject` builds, per project: a health tracker, a `thirdparty.ProvidersRegistry`, an `upstream.UpstreamsRegistry`, a `policy.Engine`, and a `NetworksRegistry` (`erpc/projects_registry.go:L123-218`). `NewNetworksRegistry` eagerly registers aliases for statically-defined networks so alias URLs work before any network bootstraps (`erpc/networks_registry.go:L67-79`). `NewNetwork` builds one `networkExecutor` per failsafe config entry plus a no-op catch-all, pre-computes the metric label (alias if set, else networkId), defaults `Architecture` to `evm` when empty, and eagerly allocates the served-tip shared counters from the shared-state registry (`erpc/networks_registry.go:L83-193`).

**chainId resolution (HTTP).** The order is: (1) **domain aliasing rules** — the first `server.aliasing.rules[]` whose `matchDomain` wildcard-matches the request Host pre-selects `projectId`/`architecture`/`chainId` (`erpc/http_server.go:L242-257`); (2) **URL path parsing** — `parseUrlPath` interprets remaining path segments against whatever is pre-selected, with a case table covering all combinations (`erpc/http_server.go:L810-1006`). Within parsing, a single unexplained segment is first tried as a **network alias** via `networksRegistry.ResolveAlias` (three call sites: nothing preselected with 2 segments `erpc/http_server.go:L860-866`; project preselected with 1 segment `L883-889`; project+architecture preselected with 1 segment `L908-923`); if alias resolution fails the segment is treated as the literal architecture/chainId. (3) **Request body fallback** — when architecture/chainId are still empty after parsing, the handler unmarshals the body and reads a top-level `"networkId"` field (e.g. `"evm:42161"`) and splits it on `:` (`erpc/http_server.go:L613-634`); otherwise the request fails with "architecture and chain must be provided in URL … or in request body … or configureed via domain aliasing" (`erpc/http_server.go:L636-642`). Finally `networkId = "<architecture>:<chainId>"` (`erpc/http_server.go:L633`) and `project.GetNetwork(httpCtx, networkId)` is called (`erpc/http_server.go:L644`). Architecture validity is checked at parse time: non-`evm` values yield "architecture is not valid (must be 'evm')" (`erpc/http_server.go:L997-999`).

**Alias registry mechanics.** `NetworksRegistry` keeps `aliasToNetworkId map[string]aliasEntry{architecture, chainID}` guarded by an RWMutex (`erpc/networks_registry.go:L30-40`). Registration happens twice: eagerly in the constructor for statically-defined networks (`L67-79`) and again in `prepareNetwork` for lazily-created networks (`L322-328`, a no-op re-registration for static ones). `registerAlias` ignores empty parts and, on conflict (same alias, different target), logs a warning and keeps the existing entry (`erpc/networks_registry.go:L334-348`). Config validation already enforces alias uniqueness per project (`common/validation.go:L610-628`) and alias charset `[a-zA-Z0-9_-]+` (`common/validation.go:L1264-1269`, `util/ids.go:L15-19`). A second, **global** alias mechanism exists for metrics only: `common.SetNetworkAliasResolver` installs an atomic `networkId → alias` function (`common/network_alias.go:L10-34`), populated at startup from all projects' static network configs (`erpc/init.go:L59-77`); components that only know the raw networkId (e.g. the gRPC cache connector at `data/grpc.go:L337`) call `common.NetworkAlias(nid)` so their metric labels match `Network.Label()`.

**Lazy network creation.** `NetworksRegistry.GetNetwork(ctx, networkId)` is the single entry point (`erpc/networks_registry.go:L214-236`): fast-path returns an already-prepared network from a `sync.Map`; otherwise it validates the id format (rejecting invalid ids with a 400 `ErrInvalidRequest` whose message suggests using an alias or `evm/42161` form, `L220-222`) and schedules a `network/<networkId>` bootstrap task on the registry's `util.Initializer`. The task (`buildNetworkBootstrapTask`, `erpc/networks_registry.go:L257-293`) does, in order: (1) `resolveNetworkConfig` — find a static `NetworkConfig` via `PreparedProject.FindNetworkConfig` or **synthesize** one: split `evm:N`, set `Architecture` + `Evm.ChainId`, run full `SetDefaults(upstreams, networkDefaults)` (so lazy networks inherit `networkDefaults` exactly like static ones), then `ExposeNetworkConfig` appends it to `project.Config.Networks` for admin-API visibility (`erpc/networks_registry.go:L350-376`, `erpc/projects.go:L58-77`); resolution failure is wrapped `common.NewTaskFatal` so the initializer never retries it (`L264-266`); (2) `upstreamsRegistry.PrepareUpstreamsForNetwork` (see below); (3) `prepareNetwork` → `NewNetwork` + cache DAL wiring + alias registration; (4) store in `preparedNetworks`; (5) `network.Bootstrap` which registers the network with the selection-policy engine (compiling a default policy if absent) and supplies the upstream list as a closure so newly-bootstrapped upstreams appear each tick (`erpc/networks.go:L107-139`). Statically-declared networks are bootstrapped through the same task machinery in background at startup (`erpc/networks_registry.go:L195-212`); a failed eager bootstrap is retried when a request arrives (`GetNetwork` re-executes the same named task — `util.Initializer` dedups by task name, `util/initializer.go:L196-215`).

**Lazy upstreams from providers.** `PrepareUpstreamsForNetwork` (`upstream/registry.go:L155-293`) schedules — exactly once per network via `sync.Once` — one `network/<networkId>/provider/<providerId>` task per configured provider, then polls every 200ms for up to 30s (capped by the caller's ctx deadline, minimum 50ms) until ≥1 upstream is registered for the network. Each provider task asks `provider.SupportsNetwork` (checks `ignoreNetworks` deny-list first, then `onlyNetworks` allow-list, then falls back to the vendor's dynamic `SupportsNetwork`, `thirdparty/provider.go:L33-52`), then `GenerateUpstreamConfigs` builds upstream configs from the provider's `overrides` (wildcard-matched on networkId) or a fresh base config, stamps the id from `upstreamIdTemplate` (`<VENDOR>`/`<PROVIDER>`/`<NETWORK>`/`<EVM_CHAIN_ID>` substitutions), sets `Type=evm` + `Evm.ChainId` from the networkId, and env-expands endpoints (`thirdparty/provider.go:L54-162`). Outcome classification on timeout: some upstream ready → success; provider tasks still pending/running → retryable `ErrNetworkInitializing` (HTTP 503, `common/errors.go:L1250-1269`); all provider tasks terminal with zero upstreams (after a 100ms grace re-check) → `ErrNetworkNotSupported` (HTTP 404, `common/errors.go:L1271-1290`) which the registry's auto-retry loop may re-attempt (`upstream/registry.go:L208-292`). Tests pin these three behaviors: slow-provider-then-serve, unsupported-network fatal, and 503-retry while initializing (`erpc/networks_bootstrap_test.go:L23-158`). The initializer that drives all of this defaults to TaskTimeout=120s, AutoRetry=true, RetryFactor=1.5, RetryMinDelay=3s, RetryMaxDelay=130s (`util/initializer.go:L175-192`); `TaskFatal`-state tasks are never retried (`util/initializer.go:L296-311`). There is also a reverse lazy path: when an upstream finishes bootstrapping, an `onUpstreamRegistered` callback calls `networksRegistry.GetNetwork` for the upstream's network and injects the network config into the upstream (`erpc/projects_registry.go:L172-183`).

**networkDefaults inheritance.** `NetworkConfig.SetDefaults(upstreams, defaults)` (`common/defaults.go:L1781-1973`) merges `project.networkDefaults` into each network with **"network wins; defaults fill gaps"** semantics; see L3 for the exact per-field rules. The most subtle part is failsafe: when both levels define failsafe lists, each network entry is matched against defaults entries by wildcard-`matchMethod` + finality compatibility and inherits from the **first** matching default (`common/defaults.go:L1786-1832`).

**Forward pipeline (network scope).** `Network.Forward` (`erpc/networks.go:L931-1603`) order: set network/cacheDal on request → apply `directiveDefaults` → start `Network.Forward` span → bind request into ctx (`common.RequestContextKey`) for served-tip selector scoping → **static responses** (`erpc/networks.go:L976-981`, exact-method + deep-param match, `common/static_response.go:L14-24`) → **multiplexer** leader/follower registration keyed by `req.CacheHash()` (`erpc/networks.go:L983-1001`, `L1989-2110`; disabled when `multiplexing: false`, `common/config.go:L2034-2039`) → **cache read** unless `skip-cache-read` directive (`L1003-1021`) → **policy-engine ordered upstream list** with cold-start fallback to registration order (`L1023-1058`) → probe-bus publish (`L1073-1075`) → EVM network pre-forward hook (`L1078-1089`) → **future-block short-circuit** (truthful `null` for concrete blocks beyond every eligible head; gated on servedTip enabled for `latest`; never cached) (`L584-610`, `L1091-1101`) → `shouldHandleMethod` (stateful-method single-upstream check; `eth_accounts`/`eth_sign` rejected with `ErrNotImplemented`) (`L2112-2143`) → network **rate-limit** permit (`L2268-2310`) → request normalization (`L1605-1630`) → failsafe-executor selection (first config-order entry matching method+finality, `L907-929`) → upstream sweep loop with per-upstream block-availability gating (`L1186-1407`), deterministic-error short-circuit (client errors / execution reverts, `L1344-1348`), emptyish-acceptance fast path (`L1314-1340`) → on failure, fall back to "last valid response" unless consensus (`L1424-1486`) → async cache set with 10s timeout + panic recovery (`L1488-1527`) → wrong-empty misbehavior accounting against block-availability bounds (`L1546-1589`).

**Finality classification.** `Network.GetFinality` (`erpc/networks.go:L1645-1743`): explicit per-method `finalized`/`realtime` flags in `methods.definitions` win; then block ref/number extracted from request, falling back to the response body (critical for hash-keyed lookups served from cache); non-numeric tags (e.g. `latest`) → `realtime`; numeric blocks are compared via the responding upstream's `EvmIsBlockFinalized`, then the request's last upstream, then the network-wide lowest finalized block heuristic; otherwise `unknown`.

**Served tip & head accessors.** Default ("max mode"): `EvmHighestLatestBlockNumber`/`EvmHighestFinalizedBlockNumber` return the MAX effective latest/finalized block across selection-policy-eligible, non-syncing upstreams (`erpc/networks.go:L508-568`, `L675-698`). Opting tags into `evm.servedTip.enabledFor` switches that axis to a cluster-min, strict-monotonic, cross-pod served tip (counters named `servedLatestBlock/<networkId>` in shared state, rollback tolerance 1024 — `erpc/networks_registry.go:L179-190`, `architecture/evm/evm_state_poller.go:L27`). Requests carrying a `use-upstream` selector get a **selector-scoped** tip: candidate upstreams are filtered to the selector's matches (`erpc/networks.go:L353-392`); a "simple" group selector (single glob token, optionally `!`-negated, matching ≥2 and <all upstreams) materializes a per-group monotonic partition keyed by the SHA-256 of the matched upstream-id set (bounded by `maxServedTipPartitions = 16`, lane-named via `common.LaneName`) (`erpc/networks.go:L405-506`, `common/lane.go:L25-74`); any other selector computes a stateless cluster-min that emits no gauges (`erpc/networks.go:L98-105`, `L802-817`). `servedTip.guaranteedMethods` clamps the global tip down to what each listed method's supporting upstreams can serve (`erpc/networks.go:L643-649`, `L700-747`). Group-selector dedup/bounds and scoped-tip semantics are pinned by `erpc/networks_served_tip_test.go:L434-537`; cluster/monotonic behavior by `L197-432`.

**Failure modes.** Invalid network id → 400; network initializing → 503 (client should retry); unsupported by all providers with no static upstreams → 404; zero upstreams after init → `ErrNoUpstreamsFound` 404 (`erpc/networks.go:L1051-1058`, `common/errors.go:L1248`); all upstreams failed → `ErrUpstreamsExhausted` carrying per-upstream errors (`erpc/networks.go:L1392-1406`).

### Behaviors & algorithms

**networkId format & validity.** Canonical: `evm:<decimal chainId>` (`util/ids.go:L11-13`). Two validators exist with different strictness: `util.IsValidNetworkId` only requires `evm:` + `strconv.Atoi`-parseable suffix (zero/negative pass) (`util/ids.go:L21-27`) and gates `GetNetwork`; `common.IsValidNetwork` additionally requires `chainId > 0` (`common/network.go:L39-49`) and gates provider `onlyNetworks`/`ignoreNetworks` validation (`common/validation.go:L800-812`).

**URL parsing decision table** (`erpc/http_server.go:L851-991`). Let P/A/C = preselected project/architecture/chain from aliasing rules. Trailing `/healthcheck` segment is stripped and flags healthcheck (`L841-849`); any non-POST/OPTIONS method is healthcheck (`L1001-1003`); `/admin` with POST/OPTIONS is the admin endpoint (`L836-838`).
- Nothing preselected: 1 seg → project only; 2 segs → project + (alias-resolve seg2, else seg2 = architecture); 3 segs → project/arch/chain; else error.
- P only: 1 seg → alias-resolve, else architecture; 2 segs → arch/chain; 3 segs → explicit override of all three.
- P+A: 1 seg → alias-resolve (must also match an arch), else chainId; 3 segs → explicit; else error.
- P+A+C: only `/` (or healthcheck) allowed.
- A+C (no P): 1 seg → project; 3 segs → explicit.
- A only: 1 seg → project; 2 segs → project/chain; 3 segs → explicit; 0 segs → healthcheck only.
- P+C without A → hard error "it is not possible to alias for project and chain WITHOUT architecture".
After the case table: project required unless healthcheck (`L993-995`); if either arch or chain ended non-empty, arch must be exactly `evm` (`L997-999`). Full matrix pinned by `TestHttpServer_ParseUrlPath`-style table tests (`erpc/http_server_test.go:L3380-3710`), including alias-in-path (`L3639-3648`).

**Alias resolution.** `ResolveAlias(alias) → (architecture, chainID)` or `("","")` under RLock (`erpc/networks_registry.go:L247-255`). Registered: (a) eagerly for every static network with non-empty `alias` whose `NetworkId()` splits into 2 parts (`erpc/networks_registry.go:L67-79`); (b) on lazy `prepareNetwork` (`L322-328`). Conflicting re-registration keeps the first entry and logs `"skipping duplicate alias registration with different target"` (`L340-345`). The alias is resolved **only in the URL path**; a body `networkId` field must be the canonical `evm:N` form (`erpc/http_server.go:L613-634`).

**GetNetwork / lazy bootstrap algorithm** (`erpc/networks_registry.go:L214-293`):
1. `preparedNetworks.Load(networkId)` hit → return.
2. `util.IsValidNetworkId` fail → `ErrInvalidRequest` (400) with hint to use alias or `evm/42161` (`L220-222`).
3. `initializer.ExecuteTasks(ctx, task("network/<id>"))` — task dedup by name; caller waits on its own ctx while the task runs on appCtx (`util/initializer.go:L196-215`, `L248-341`).
4. Task body: `resolveNetworkConfig` (fatal on bad format → no retries, `L264-266`; synthesizes config + `SetDefaults` + `ExposeNetworkConfig` for unknown ids, `L350-376`) → `PrepareUpstreamsForNetwork` (provider fan-out + ready-wait; logs `"network initialization ended with zero upstreams"` on empty result, `L268-279`) → `prepareNetwork` (idempotent via preparedNetworks check; wires cache DAL per architecture; unknown architecture → `errors.New("unknown network architecture")`, `L295-330`) → store → `network.Bootstrap` (policy-engine registration, `erpc/networks.go:L113-139`).
5. After task completion, missing entry in `preparedNetworks` → `ErrNetworkNotFound` (404) (`L230-235`).
Initializer auto-retries Failed/TimedOut tasks with backoff (factor 1.5, 3s..130s; task timeout 120s) but never Fatal ones (`util/initializer.go:L175-192`, `L296-311`).

**Provider readiness state machine** (`upstream/registry.go:L188-292`): wait up to min(30s, ctx deadline) polling at 200ms; `summarizeNetworkTasks` streams task states by name-prefix `network/<id>/provider/` and `network/<id>/upstream/` (`L529-562`). ≥1 upstream → nil; ongoing tasks → `ErrNetworkInitializing` (503); all providers terminal + still zero upstreams after 100ms grace → `ErrNetworkNotSupported` (404, retryable by auto-retry loop). Provider task: `SupportsNetwork` (ignoreNetworks → onlyNetworks → vendor dynamic check) then `GenerateUpstreamConfigs` then `registerUpstreams` (`upstream/registry.go:L477-510`, `thirdparty/provider.go:L33-65`).

**Block-availability enforcement precedence and tri-state semantics** (`resolveEnforceBlockAvailability`, `erpc/networks.go:L1786-1831`):

`enforceBlockAvailability` is a `*bool` at both the method level and the network level. The three pointer states have distinct meanings:

| Value | Meaning |
|---|---|
| `nil` (unset) | Defer to the next lower-priority source in the chain. At the method level this means "not a user override". At the network level this means "fall through to per-upstream bounds or system default". |
| `true` | Enforcement explicitly enabled (block-unavailable checks run for this method/network). |
| `false` | Enforcement explicitly disabled (block-unavailable checks are skipped entirely, even if an upstream has `blockAvailability` bounds configured). |

Note that `nil` at the network level does NOT mean "enabled" — it means "unset; defer". Enforcement is only enabled at that step if the pointer is non-nil and `*ptr == true`. The fallback (step 5) resolves to `true` only when every higher-priority step yields no decision.

**Full 5-step precedence chain** (evaluated in order; first non-nil decision wins):

1. **Method-level override that differs from the system default** (`erpc/networks.go:L1806-1813`). `MethodsConfig.SetDefaults` merges `DefaultWithBlockCacheMethods` into `Methods.Definitions`, so the system default value is present in `n.cfg.Methods.Definitions[method].EnforceBlockAvailability`. A user's method-level value is only treated as an override when it differs from that merged-in system default (pointer-equality comparison of the bool value). If the user explicitly sets the same value as the system default, it is treated as not-an-override and step 1 falls through. This is the only mechanism allowing per-method enforcement to differ from the network-level setting.
2. **Network-level `evm.enforceBlockAvailability`** (`erpc/networks.go:L1814-1817`). Non-nil pointer wins regardless of value. `nil` falls through.
3. **Upstream has `blockAvailability` bounds** (`erpc/networks.go:L1818-1824`). If the current upstream (`u`) has `Evm.BlockAvailability.Lower != nil || Upper != nil`, enforcement is forced **on** (`return true`) regardless of system defaults. This ensures users who configure bounds actually get them enforced. `false` at step 2 short-circuits before reaching step 3.
4. **System default from `DefaultWithBlockCacheMethods[method].EnforceBlockAvailability`** (`erpc/networks.go:L1825-1828`). Non-nil system default wins. `nil` (method not in `DefaultWithBlockCacheMethods`) falls through.
5. **Fallback: enabled** (`erpc/networks.go:L1830`). `return true`.

**Inconsistent documentation across KB files** (authoritative resolution): census §7 and some KB entries say "nil = enforcement enabled" but this is imprecise. `nil` at the network level means "defer"; enforcement happens to be enabled as the fallback only when *all five steps* yield no decision. Setting `evm.enforceBlockAvailability: null` (nil) is different from setting it to `false` — the former defers to upstream bounds and system defaults; the latter disables enforcement even when an upstream has explicit bounds.

**Pre-canned upstream templates** (e.g. in `thirdparty/`) typically ship with `EnforceBlockAvailability` unset (nil) unless they have specific bounds, so they defer to the standard chain. The overall fallback (step 5) means the effective default for any method not in `DefaultWithBlockCacheMethods` is **enabled**.

The check itself fails open on extraction/poller errors (`erpc/networks.go:L1919-1959`); not-available results are classified retryable iff `bn - latestBlock <= maxRetryableBlockDistance` (default 128) (`erpc/networks.go:L1960-1985`).

**GetFinality algorithm** (`erpc/networks.go:L1645-1743`): method config `finalized` → Finalized; `realtime` → Realtime; else extract blockRef/blockNumber from request, then from response (cache-hit path); blockNumber==0 and ref empty/`*` → Unknown; non-numeric ref → Realtime; numeric → `EvmIsBlockFinalized` via response upstream, then request's last upstream, then network `EvmLowestFinalizedBlockNumber` heuristic (span attr `used_network_finalized_heuristic=true`); else Unknown. Enum: Finalized=0, Unfinalized=1, Realtime=2, Unknown=3 (`common/data.go:L8-26`).

**Served-tip computation** (`erpc/networks.go:L508-852`, picker in `architecture/evm/served_tip.go:L114`): per axis — candidates = policy-eligible upstreams for `*` (or selector subset), excluding non-EVM/no-poller/syncing/zero-block (`L317-347`); `ComputeServedTipCandidate(tips, lastServed, elapsed, cfg)` with EMA block time + clusterDelta; `method=="*"` candidate clamped by `guaranteedMethodFloor`; monotonic `TryUpdate` on the shared counter (rollback tolerance `evm.DefaultToleratedBlockHeadRollback = 1024`, `architecture/evm/evm_state_poller.go:L27`); served value = post-clamp counter value. Partition materialization rules: `isSimpleGroupSelector` (≤128 chars, no whitespace/boolean chars, at most one leading `!`, `L493-506`); matched set must be ≥2 and < all (`L466-486`); cap 16 partitions (`L84`); key = `grp:` + first 8 bytes hex of SHA-256 over sorted ids joined by `\x00` (`L483-485`). `EvmLeaderUpstream` = upstream with max raw `LatestBlock()` (`L891-905`); `EvmLowestFinalizedBlockNumber` = min positive effective finalized across non-syncing upstreams (`L854-889`).

**State-poller enrichment.** Successful `eth_getBlockByNumber` with tag `latest`/`finalized` (tag recovered from preserved `EvmBlockRef` even after hex interpolation, falling back to raw params) suggests the response's block number to the serving upstream's poller; `eth_blockNumber` suggests latest (`erpc/networks.go:L2145-2232`).

**Response ID fidelity.** `normalizeResponse` overwrites the upstream-echoed JSON-RPC id with the client's original — preferring verbatim raw id bytes to round-trip >2^53 integers and exotic formats (`erpc/networks.go:L2234-2266`; tests `erpc/networks_normalize_response_id_fidelity_test.go`).

**Error codes produced at this layer.** `ErrInvalidRequest` (400, bad networkId format `erpc/networks_registry.go:L221`); `ErrNetworkNotFound` (404, `common/errors.go:L609-627`); `ErrNetworkInitializing` (503, `common/errors.go:L1250-1269`); `ErrNetworkNotSupported` (404, `common/errors.go:L1271-1290`); `ErrInvalidEvmChainId` (400, `common/errors.go:L680-696`); `ErrNoUpstreamsFound` (404, `common/errors.go:L1248`); `ErrUpstreamsExhausted` (`erpc/networks.go:L1392-1404`); `ErrNotImplemented` (501, `common/errors.go:L665-678`); `ErrNetworkRateLimitRuleExceeded` (`erpc/networks.go:L2299-2305`); `ErrNetworkRequestTimeout` (see dedicated entry below); `ErrInvalidUrlPath` variants from parsing (`erpc/http_server.go:L873-994`); `ErrUnknownNetworkID`/`ErrUnknownNetworkArchitecture` (400, `common/errors.go:L629-663`).

**`ErrNetworkRequestTimeout` — full semantics** (`common/errors.go:L1783-1799`, produced at `erpc/networks.go:L2079-2084`):

This error is produced exclusively in the **multiplex-follower path**: a follower goroutine waiting on the leader's `done` channel hits its own context deadline before the leader completes (`erpc/networks.go:L2079-2082`):

```go
case <-ctx.Done():
    err := ctx.Err()
    if errors.Is(err, context.DeadlineExceeded) {
        return nil, common.NewErrNetworkRequestTimeout(time.Since(startTime), err)
    }
```

It is **not** the same as `ErrFailsafeTimeoutExceeded` (which is produced by the failsafe policy engine when its configured `timeout` policy fires during the upstream sweep, `common/errors.go:L1586-1612`). The distinction:

| Error | Source | Scope |
|---|---|---|
| `ErrNetworkRequestTimeout` | Multiplexer follower ctx deadline | Network-scope — the follower's request-context timed out waiting for the shared in-flight result |
| `ErrFailsafeTimeoutExceeded` | Failsafe policy engine | Failsafe policy `timeout` field exceeded during the upstream execution loop |
| `ErrEndpointRequestTimeout` | HTTP client round-trip | Upstream-scope — individual upstream call timed out |

**Retryability** (from `IsRetryableTowardNetwork` / `IsRetryableTowardsUpstream`, `common/errors.go:L2379-2497`): `ErrNetworkRequestTimeout` carries no explicit `retryableTowardNetwork: false` in its `Details` map, so `IsRetryableTowardNetwork` returns `true` (default) and `IsRetryableTowardsUpstream` also returns `true` (not in the explicit non-retryable list). Census notation: U:yes N:yes.

**Wire behavior**: `ErrorStatusCode()` returns `http.StatusGatewayTimeout` (504) (`common/errors.go:L1797-1799`). However, `ErrNetworkRequestTimeout` is NOT in the HTTP server's 404/401/429 status-code lists (`erpc/http_server.go:L1280-1317`), so it falls through to the default 200 response body path for POST requests — the method-level status is 504 but the wire transport status is 200 (JSON-RPC over HTTP keeps 200). The JSON-RPC error code is −32603.

**Message format**: `"network-level request towards one or more upstreams timed out after <N>ms"` (`common/errors.go:L1791`). The `N` is the elapsed duration since the follower registered with the multiplexer (`erpc/networks.go:L2034-2038`, `startTime := time.Now()`).

### Edge cases & gotchas

1. **Aliases work in URLs only, not in body `networkId`** — body fallback splits the literal string on `:` with no alias lookup (`erpc/http_server.go:L613-634`); alias resolution happens exclusively inside `parseUrlPath` (`erpc/http_server.go:L860-923`).
2. **`GetNetwork` rejects aliases**: by the time it runs, the id must be `evm:<int>`; the 400 error message explicitly tells callers to use `/main/arbitrum` or `/main/evm/42161` (`erpc/networks_registry.go:L220-222`).
3. **Alias resolution failure is silent**: an unknown single path segment falls through to being treated as architecture/chainId, producing "architecture is not valid (must be 'evm')" rather than an "unknown alias" error (`erpc/http_server.go:L860-866`, `L997-999`).
4. **Aliases for lazy networks only register after first use/bootstrap** of that network (`erpc/networks_registry.go:L322-328`); only statically-declared networks get eager constructor-time registration (`L67-79`). A config with alias but no `evm` block yields `NetworkId() == ""` (`common/config.go:L2592-2603`) and the alias is skipped (split-length guard `L72-75`).
5. **The metrics alias resolver is static-config only**: `erpc/init.go:L62-77` builds the map from `cfg.Projects[].Networks[]` at startup; lazily-exposed networks never get entries, and `common.NetworkAlias` then returns the raw networkId (`common/network_alias.go:L27-34`) — gRPC-cache metrics can diverge from request metrics for lazy networks.
6. **`NetworkConfig.evm.chainId` is never validated > 0** (`common/validation.go:L1302-1330`); `evm:0` also passes `util.IsValidNetworkId` (`util/ids.go:L21-27`) while provider network filters use the stricter `chainId > 0` check (`common/network.go:L39-49`).
7. **Stale comment trap**: `NewNetwork` says an empty label "will render as n/a via Network.Label()" (`erpc/networks_registry.go:L144`) but `Label()` actually falls back to the networkId (`erpc/networks.go:L278-286`); only a nil receiver or nil request yields `""`/`"n/a"` (`common/request.go:L511-525`).
8. **Legacy single-object `networkDefaults.failsafe` silently drops `multiplexing`**: the old-format fallback struct has no `Multiplexing` field, so that key is lost when the legacy decode path triggers (`common/config.go:L626-656`).
9. **`directiveDefaults` is all-or-nothing across levels**: a network-level block completely replaces `networkDefaults.directiveDefaults` (nil-check copy only, `common/defaults.go:L1837-1840`) — individual directive fields are NOT merged.
10. **Failsafe merge picks the FIRST matching default** (`break` at `common/defaults.go:L1813-1816`); ordering of `networkDefaults.failsafe` entries matters when multiple wildcard entries could match.
11. **A failed lazy bootstrap is user-retryable, not poisoned**: failed (non-fatal) tasks are retried both by the background auto-retry loop and by the next request's `ExecuteTasks` (`erpc/networks_registry.go:L225-229`, `util/initializer.go:L248-341`); only `resolveNetworkConfig` format errors are `TaskFatal` and never retried (`erpc/networks_registry.go:L263-266`). 503-on-initializing and fatal-on-unsupported are pinned by `erpc/networks_bootstrap_test.go:L83-158`.
12. **Provider `onlyNetworks` bypasses vendor support checks entirely** — a networkId listed there returns true without consulting the vendor; one not listed returns false (`thirdparty/provider.go:L42-49`); `ignoreNetworks` is evaluated first (`L34-40`).
13. **Synthesized lazy configs become permanently visible**: `ExposeNetworkConfig` appends to `project.Config.Networks` (admin API surface) and never overwrites an existing entry (`erpc/projects.go:L58-77`).
14. **Stateful methods + multiple upstreams = 501**: without a single-upstream `use-upstream` directive, any network with >1 upstream rejects `eth_newFilter`-style methods (`erpc/networks.go:L2112-2136`); `eth_accounts`/`eth_sign` are always rejected (`L2138-2140`).
15. **Multiplexer follower-of-closed-leader race loops to leadership**: a follower that finds `closed=true` retries `LoadOrStore` to become the new leader (`erpc/networks.go:L2003-2023`); cleanup blocks the leader goroutine until all followers finish copying (`L2088-2110`). Thundering-herd dedup (1 upstream call for N concurrent identical requests) pinned by `erpc/networks_multiplexer_test.go:L34-300` and `TestNetwork_ThunderingHerdProtection` (`erpc/networks_test.go:L10434`).
16. **Syncing upstreams are excluded from every head computation** — max mode, cluster mode, lowest-finalized, and guaranteed-method floors all skip `EvmSyncingStateSyncing` (`erpc/networks.go:L327-330`, `L554-557`, `L719-721`, `L870-874`; test `erpc/networks_test.go:L11252-11375`).
17. **Selector-scoped tips never pollute network gauges**: stateless scoped picks use lane sentinel `"\x00scoped"` and emit nothing (`erpc/networks.go:L98-105`, `L806-810`); equivalent selectors dedup into one partition keyed by matched-set hash, garbage/boolean/single-node/match-all selectors create no state, cap 16 (`erpc/networks_served_tip_test.go:L488-536`).
18. **Future-block short-circuit compares against MAX head, not the cluster-min served tip**, so it never nulls a block the most-ahead upstream has; it requires servedTip enabled for `latest` and a concrete block number (tags/hashes never short-circuit) and the synthetic null is never cached (`erpc/networks.go:L570-610`).
19. **Wrong-empty misbehavior is suppressed for out-of-bounds blocks**: if the response block number falls outside an upstream's configured availability bounds, its empty response is expected and no misbehavior is recorded (`erpc/networks.go:L1566-1587`).
20. **`evm.getLogsMaxAllowedAddresses`/`Topics` default to 0 = unlimited** (no SetDefaults entry; enforcement guarded by `> 0`, `architecture/evm/eth_getLogs.go:L212-215`) while `getLogsMaxAllowedRange` defaults to 30000 and must be > 0 (`common/defaults.go:L2092-2094`, `common/validation.go:L1309-1311`) — an explicit `getLogsMaxAllowedRange: 0` is impossible (validation fails), but inheriting networks treating 0 as "unset" also means you cannot explicitly zero-out a default-bearing numeric field.
21. **Two `RequestContextKey` binds in Forward**: once early for served-tip/pre-forward paths (`erpc/networks.go:L956`) and once for the failsafe executor ctx (`L1170`) — both intentional, same value.
22. **`ErrNetworkNotFound` after successful task** is possible only if the task stored nothing (defensive path `erpc/networks_registry.go:L230-235`); the doc comment notes first-attempt failures must propagate so the user retries.
23. **Zero projects → implicit `main` project + catch-all aliasing rule** (`matchDomain: "*"`, `serveProject: "main"`), so `/evm/123` works without a project segment (`common/defaults.go:L100-110`).
24. **`stateful: false` on a built-in stateful method is silently overridden** — `MethodsConfig.SetDefaults` iterates `DefaultStatefulMethodNames` and force-sets `cm.Stateful = true` in all three code paths (no-definitions, `preserveDefaultMethods: true`, custom-only), overwriting any user-supplied `stateful: false` without any warning or validation error (`common/defaults.go:L512-519`, `L546-553`, `L562-573`). The six affected methods are: `eth_newFilter`, `eth_newBlockFilter`, `eth_newPendingTransactionFilter`, `eth_getFilterChanges`, `eth_getFilterLogs`, `eth_uninstallFilter`. This also means these six methods are injected into `Definitions` even if the user set `preserveDefaultMethods: false` and did not mention them — they always appear.
25. **`enforceBlockAvailability: false` at network level overrides upstream bounds** — step 3 of the enforcement chain (upstream-has-bounds) is only reached when steps 1 and 2 both yield nil. An explicit `evm.enforceBlockAvailability: false` at the network level (step 2) short-circuits before the upstream-bounds check, so users who configure `blockAvailability` on upstreams but set `enforceBlockAvailability: false` at the network level get no enforcement (`erpc/networks.go:L1814-1817`).
26. **`ErrNetworkRequestTimeout` is produced only in the multiplexer follower path**, not during the upstream sweep or failsafe execution. A request that is itself the multiplexer leader experiences `ErrFailsafeTimeoutExceeded` (or the upstream sweep's own errors) if it times out, never `ErrNetworkRequestTimeout` (`erpc/networks.go:L2079-2084` vs `erpc/networks.go:L1424-1452`).
27. **`networkDefaults.multiplexing` does not deep-merge with network-level multiplexing** — it is a flat `*bool` copy; there is no way to enable multiplexing for specific networks and disable for others via the defaults layer alone. A network must set its own `multiplexing: false` to deviate from the project-wide default.
28. **`EnforceGetLogsBlockRange` and `EnforceHighestBlock` directives cannot be overridden per-request via HTTP headers for `eth_getLogs`, `trace_filter`, `arbtrace_filter`, and `eth_blockNumber`** — the architecture hooks for these methods read from the legacy `ncfg.Evm.Integrity` struct, NOT from per-request directives. `eth_getLogs.go:L282-285` and `trace_filter.go:L289-292` check `ncfg.Evm.Integrity.EnforceGetLogsBlockRange`, and `eth_blockNumber.go:L23-27` checks `ncfg.Evm.Integrity.EnforceHighestBlock`. Both are populated from `directiveDefaults` at `SetDefaults` time (`common/defaults.go:L1957-1964`), so they reflect the network config value baked in at startup — sending `X-ERPC-Enforce-GetLogs-Range: false` or `X-ERPC-Enforce-Highest-Block: false` in an HTTP header has no effect on these methods. (By contrast, `eth_getBlockByNumber.go:L118` DOES check `dirs.EnforceHighestBlock` from the live per-request directives — so the override works there.) This limitation means `enforceGetLogsBlockRange: false` must be set in the network's `directiveDefaults` config, not sent per-request, to disable the range check.
29. **`preserveDefaultMethods: false` with NO `definitions` block keeps ALL built-ins** — the condition at `common/defaults.go:L494` is `len(m.Definitions) == 0 && !m.PreserveDefaultMethods`, which evaluates to true and uses all defaults. The "REPLACE" behavior only triggers when definitions are non-empty AND `preserveDefaultMethods` is false. A config with `methods: { preserveDefaultMethods: false }` and no `definitions` key is therefore identical to omitting the `methods` block entirely.
30. **`finalized: true` on a non-finalized method permanently caches all responses** — once `GetFinality` returns `DataFinalityStateFinalized` for a method, the cache subsystem places every response in the `finalized` bucket (long/permanent TTL). There is no per-response override: a method flagged `finalized: true` has ALL its responses cached as if immutable, even if they happen to contain data from an unfinalized block. This is appropriate for `eth_chainId` (always immutable) but catastrophic for, e.g., `eth_getBalance` — every balance query would be permanently cached at the first value ever returned (`erpc/networks.go:L1656-1660`).
31. **`translateLatestTag: false` on a custom method config does not prevent cache key deduplication breaking** — when `nil`/`true`, two requests with `"latest"` that resolve to different block numbers at different times get different cache keys (the hex is substituted). When `false`, both get the same `"latest"` cache key, so the second request is served from cache regardless of the current block number. This can cause stale responses for rapidly-changing data if cache TTL exceeds block time and `translateLatestTag: false` is set on a non-historical method.
