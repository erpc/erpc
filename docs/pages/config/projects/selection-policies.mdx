# Selection policy

A single JavaScript function (`eval`) per network returns the **ordered**
list of upstreams that should serve requests. The order IS the routing
decision: position 0 is the primary, position N is the Nth retry,
anything missing is excluded.

<a id="priority--routing"></a>

<Callout type="info">

</Callout>

## How it works

```
                              every `evalInterval`
                                       ▼
upstreams + ctx (cross-tick state) ──► eval() ──► ordered upstream[]
                                                       │
                                                       ▼
                            Network.Forward consumes the list on every
                            request (wait-free atomic load, O(1))
```

- The eval runs on a per-(network, method) timer.
- Output is cached atomically; the request path reads it lock-free.
- Every tick produces a **decision record** joined to requests via
  `decision_id` — `GET /admin/selection/explain/<requestId>` answers
  *"why was upstream X used at time T for method M?"*.

| Field | Type | Default | Notes |
|---|---|---|---|
| `evalInterval` | Duration | `1s` | `0` disables the ticker (test only). |
| `evalTimeout` | Duration | `100ms` | Must be `< evalInterval`. Prior cache retained on timeout. |
| `evalPerMethod` | bool | `false` | Separate eval + cache per method. |
| `decisionHistory` | Duration | `5m` | Ring buffer = `decisionHistory / evalInterval`. |
| `eval` | string | (built-in default) | JS function returning `Upstream[]`. |

## The default policy

Omitting `eval` applies a conservative, chain-agnostic policy designed
for production:

```js
(upstreams, ctx) =>
  upstreams
    .removeCordoned()                  // 1. drop failsafe-cordoned
    .removeByErrorRate(0.8)            // 2. drop > 80% errors over window
    .whenEmpty(() => upstreams)        // 3. safety net: never empty
    .preferGroup('!fallback', {        // 4. primary tier = group !== 'fallback'
      minHealthy: 1,
      fallback: 'fallback',
    })
    .sortByScore(BALANCED)             // 5. rank by composite score
    .stickyPrimary({                   // 6. anti-flap
      hysteresis: 0.10,
      minSwitchInterval: '30s',
    })
    .probeExcluded({                   // 7. recover excluded upstreams
      reAdmitAfter: '5m',
      maxConcurrent: 1,
    })
```

Step-by-step:

1. **Drop cordoned** — failsafe / circuit-breaker explicitly marked the upstream bad.
2. **Drop clearly broken** — 80% errors over the window means broken on any chain.
3. **Safety net** — if filtering wiped everything (network-wide outage), serve from the raw set rather than fail closed.
4. **Tier** — `group !== 'fallback'` is the primary tier; `group === 'fallback'` is the second tier. Matches the long-standing eRPC convention so adding a fallback upstream is one config line.
5. **Score** — BALANCED weights: `errorRate=8, respLatency=4, throttledRate=3, blockHeadLag=2, finalizationLag=1, misbehaviors=6`. Lag and latency are CHAIN-SPECIFIC and contribute to ranking, not filtering, so the default works on Ethereum, Arbitrum, and everything in between.
6. **Sticky primary** — keep the current primary unless a challenger's score is ≥10% better AND 30s passed since the last switch. Prevents flapping under transient blips.
7. **Probe excluded** — every tick, re-admit one upstream that has been out for 5+ minutes so its metrics refresh.

Source served at `GET /admin/selection/default-policy`.

### Production-tuned alternative

The default ships *conservative* on purpose — it won't surprise users
with aggressive exclusion. For high-traffic environments running tight
finality SLOs, the same primitives compose into a stricter chain that
addresses the most common operational failure classes (head-lag rotation,
slow-upstream drag, prolonged-bad-primary during reorgs):

```js
(upstreams, ctx) =>
  upstreams
    .removeCordoned()
    .keepHealthy({                                  // ← stricter filter
      maxErrorRate: 0.5,
      maxBlockHeadLag: 16,
      maxP95Ms: 10_000,
      maxThrottledRate: 0.3,
    })
    .removeByLag({ blockHead: 16, finalization: 64 })
    .whenEmpty(() => upstreams)
    .preferGroup('!fallback', { minHealthy: 1, fallback: 'fallback' })
    .sortByScore(BALANCED)
    .if(ctx.finality === REALTIME,                  // ← no sticky on finalized
        (u) => u.stickyPrimary({ hysteresis: 0.10, minSwitchInterval: '30s' }),
        (u) => u)
    .probeExcluded({ reAdmitAfter: '90s', maxConcurrent: 2 })
```

The four differences vs. the conservative default:
- **filter, don't just rank** lag and p95 latency before sorting,
- **tighter error threshold** (`0.5` vs `0.8`),
- **drop stickiness for finalized/unfinalized** requests (reorg-tolerant anyway),
- **faster probe re-admit** (`90s` vs `5m`) for transient-blip recovery.

If your operators have ever had to comment out an upstream because
it was lagging but error-free, the chain above is what you want.

## Eval inputs

Inside the function body, two variables are in scope: `upstreams` and `ctx`.

```ts
type Upstream = {
  readonly id: string
  readonly vendor: string                  // "alchemy", "infura", "drpc"
  readonly type: 'evm' | string
  readonly endpoint: string                // redacted

  readonly config: UpstreamConfig          // user config (group, etc.)
  readonly metrics: UpstreamMetrics        // tick-start snapshot

  // Attached by std-lib steps:
  readonly score?: number                  // sortByScore (lower = better)
  readonly penaltyBreakdown?: ScoreBreakdown
  readonly annotations?: string[]
}

type UpstreamMetrics = {
  errorRate: number                        // 0..1
  errorsTotal: number
  requestsTotal: number
  throttledRate: number                    // 0..1
  misbehaviorRate: number
  p50ResponseSeconds: number
  p70ResponseSeconds: number
  p90ResponseSeconds: number
  p95ResponseSeconds: number
  p99ResponseSeconds: number
  blockHeadLag: number                     // BLOCK-number delta, NOT seconds
  finalizationLag: number                  // block-number delta
  cordonedReason: string | null
}

type EvalContext = {
  network: string                          // "evm:1"
  method: string                           // "*" unless evalPerMethod=true
  finality: 'realtime' | 'unfinalized' | 'finalized' | 'unknown'
  now: number                              // unix ms

  // Cross-tick state — the ONLY carrier of state between ticks.
  previousOrder: string[]
  previousExcluded: string[]
  lastSwitchAt: number | null
  excludedSince: { [id: string]: number }
  tickCount: number
}
```

The `metrics` object is captured ONCE at tick start, so chained std-lib
steps see a consistent view.

Globals: `BALANCED`, `PREFER_FASTER`, `PREFER_FEWER_ERRORS`,
`PREFER_FRESHER_HEAD`, `PREFER_LESS_THROTTLED`, `PREFER_CHEAP`;
`REALTIME`/`UNFINALIZED`/`FINALIZED`/`UNKNOWN`;
`REASON_*` reason codes; `process.env`; `console.log/info/warn/error`;
standard ECMAScript.

## Common patterns

### Cost-tier routing
```js
// Cheap pool first; fall through to fast pool when cheap is exhausted.
(upstreams, ctx) =>
  upstreams
    .removeByErrorRate(0.5)
    .preferGroup('cheap', { minHealthy: 2, fallback: 'fast' })
    .sortByScore(BALANCED)
```

### Per-method routing
```js
(upstreams, ctx) => {
  if (methodMatches(['eth_getLogs', 'eth_getBlockByNumber']))
    return upstreams.byGroup('archive').sortByScore(PREFER_FASTER)
  return upstreams.sortByScore(BALANCED).stickyPrimary()
}
```

### Per-upstream weights (legacy `scoreMultipliers` replacement)
```js
const w = {
  alchemy: { errorRate: 8, respLatency: 12 },
  drpc:    { errorRate: 8, respLatency: 4 },
}
(upstreams, ctx) =>
  upstreams.sortByScore((u) => w[u.vendor] || BALANCED)
```

More patterns + every method explained at
[selection policy stdlib](/config/projects/selection-policy-stdlib#examples).

## Observability

| Surface | What it gives you |
|---|---|
| `GET /admin/selection/<net>/<method>` | Latest decision (input metrics, eval duration, output order, exclusions, diff vs prior tick) |
| `GET /admin/selection/<net>/<method>?since=10m` | Sparse log of recent decisions |
| `GET /admin/selection/explain/<requestId>` | Join a request to the decision active at its time |
| `GET /admin/selection/default-policy` | Source of the built-in default |
| `POST /admin/selection/<net>/<method>/reeval` | Force a tick |
| `erpc_selection_position{...}` (gauge) | 0 = primary, 1+ = runner-up, -1 = excluded this tick |
| `erpc_selection_rejection_total{step, ...}` | Per-std-lib-step rejections |
| `erpc_selection_primary_switch_total{from, to, ...}` | Primary changes |
| `erpc_selection_eval_duration_seconds{...}` | Per-tick eval latency |
| `erpc_selection_eligible_upstreams{...}` | Count returned by the most recent tick |

Every request log includes the `decision_id` of the decision active when
it was served — pair with `/explain/<requestId>` for full attribution.
