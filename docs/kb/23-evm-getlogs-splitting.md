## L1 — Capability (CTO view)

eRPC transparently intercepts `eth_getLogs` requests and can automatically split oversized queries into smaller parallel sub-requests before forwarding, then merge and return the concatenated results to the caller. This prevents individual upstreams from rejecting large block-range requests with 413-style errors, eliminating a common source of client-side failures and making the proxy usable as a universal logs archive gateway. Two complementary paths exist: proactive pre-forward splitting (before any upstream sees the request) and reactive post-forward splitting (triggered by upstream error responses). Both paths use identical concurrent fan-out and deterministic ordered merge mechanics.

## L2 — Mechanics (staff-engineer view)

### Hook registration

Four hooks are registered in `architecture/evm/hooks.go` for `eth_getLogs`:

1. **Project pre-forward** (`HandleProjectPreForward` → `projectPreForward_eth_getLogs`): fires before cache lookup and upstream selection. It only emits a metrics observation of the raw block-range size; it never short-circuits or modifies the request. (`architecture/evm/hooks.go:L29-L30`, `architecture/evm/eth_getLogs.go:L83-L129`)

2. **Network pre-forward** (`HandleNetworkPreForward` → `networkPreForward_eth_getLogs`): fires after upstream selection, before the request is forwarded. This is where proactive splitting happens. It resolves block tags, validates range/address/topic hard limits, and — if `GetLogsAutoSplittingRangeThreshold` is exceeded — generates contiguous block sub-requests and executes them concurrently, returning a merged response without reaching the upstream for the parent request. (`architecture/evm/hooks.go:L50-L51`, `architecture/evm/eth_getLogs.go:L135-L271`)

3. **Upstream pre-forward** (`HandleUpstreamPreForward` → `upstreamPreForward_eth_getLogs`): fires for each upstream candidate. If `EnforceGetLogsBlockRange` (under `directiveDefaults` or the legacy `evm.integrity`) is enabled, it calls `CheckBlockRangeAvailability` to verify both `fromBlock` and `toBlock` are available on that specific upstream before sending. (`architecture/evm/hooks.go:L96-L97`, `architecture/evm/eth_getLogs.go:L273-L347`)

4. **Network post-forward** (`HandleNetworkPostForward` → `networkPostForward_eth_getLogs`): fires after the upstream returned an error. If `GetLogsSplitOnError` is enabled and the error is `ErrCodeEndpointRequestTooLarge` or `JsonRpcErrorEvmLargeRange` (`-32012`), it invokes `splitEthGetLogsRequest` and retries the split sub-requests. (`architecture/evm/hooks.go:L75-L76`, `architecture/evm/eth_getLogs.go:L364-L412`)

5. **Upstream post-forward** (`HandleUpstreamPostForward` → `upstreamPostForward_eth_getLogs`): if the upstream returned success with an empty/null result, it normalizes the response to a canonical `[]` array using `normalizeEmptyArrayResponse`. (`architecture/evm/hooks.go:L137-L138`, `architecture/evm/eth_getLogs.go:L349-L362`)

### Re-entrancy guard

All split-producing hooks check for `nrq.ParentRequestId() != nil || nrq.IsCompositeRequest()` and return `(false, nil, nil)` immediately. This prevents infinite recursion when sub-requests generated by a split flow back through the same hook chain. (`architecture/evm/eth_getLogs.go:L141-L143`, `architecture/evm/eth_getLogs.go:L373-L374`)

### Proactive splitting flow (network pre-forward)

1. If either `fromBlock` or `toBlock` cannot be resolved to a numeric block (tags `safe`, `pending`, unresolvable `earliest`, or no state available), the request passes through to the upstream unmodified — block tags `latest` and `finalized` are resolved via `resolveBlockTagToHex` using the highest known block from the network's state poller. (`architecture/evm/eth_getLogs.go:L185-L194`, `architecture/evm/json_rpc.go:L55-L83`)
2. If `fromBlock > toBlock`, returns `ErrInvalidRequest` immediately. (`architecture/evm/eth_getLogs.go:L196-L200`)
3. Hard limit checks (in order): `GetLogsMaxAllowedRange`, `GetLogsMaxAllowedAddresses`, `GetLogsMaxAllowedTopics`. These fire before auto-splitting and return 413 HTTP to the client. (`architecture/evm/eth_getLogs.go:L208-L217`)
4. Effective threshold: the minimum `GetLogsAutoSplittingRangeThreshold` across all selected upstreams that have a positive value. Upstreams with zero/unconfigured threshold are ignored. If the result exceeds `GetLogsMaxAllowedRange`, it is capped to that value. (`architecture/evm/eth_getLogs.go:L220-L237`)
5. If `requestRange > effectiveThreshold`, generates `ceil(requestRange/effectiveThreshold)` contiguous sub-requests of at most `effectiveThreshold` blocks each: `[from, from+T-1], [from+T, from+2T-1], …, [last_start, toBlock]`. The last chunk may be smaller. (`architecture/evm/eth_getLogs.go:L241-L253`)
6. Sets composite type `CompositeTypeLogsSplitProactive` on the parent request, then calls `executeGetLogsSubRequests`. (`architecture/evm/eth_getLogs.go:L255-L267`)

### Reactive splitting flow (network post-forward / split-on-error)

Triggered only when: (a) `GetLogsSplitOnError = true`, (b) the parent request has no `ParentRequestId` and is not already a composite, and (c) the upstream error is `ErrCodeEndpointRequestTooLarge` or `ErrJsonRpcExceptionInternal` with `NormalizedCode() == JsonRpcErrorEvmLargeRange` (`-32012`). (`architecture/evm/eth_getLogs.go:L370-L391`)

`splitEthGetLogsRequest` applies a priority hierarchy:
1. **Block range > 1**: bisect the block range at the midpoint (`mid = fb + blockRange/2`). The left chunk is `[fb, mid-1]` and the right chunk is `[mid, tb]`. There is **no configurable chunk size** — the reactive path always bisects, meaning chunk sizes are `floor(N/2)` and `ceil(N/2)`. (`architecture/evm/eth_getLogs.go:L563-L577`)
2. **Single block, multiple addresses**: bisect the address list. (`architecture/evm/eth_getLogs.go:L580-L597`)
3. **Single block, single/no address, `topics[0]` is an OR-list (array) with > 1 entry**: bisect `topics[0]`. (`architecture/evm/eth_getLogs.go:L599-L623`)
4. **Cannot split further**: returns an error; the original error is propagated to the caller unchanged. (`architecture/evm/eth_getLogs.go:L625`)

This splitting is recursive: sub-requests themselves go through the full network forward path, so if a sub-request also gets a too-large error and `GetLogsSplitOnError` is enabled, it will be split again — down to a single block, then by address, then by topic. Test `TestExecuteGetLogsSubRequests_WithNestedSplits` exercises this three-level recursion. (`architecture/evm/eth_getLogs_test.go:L1052-L1157`)

### Proactive vs. reactive splitting: interaction and precedence

The two splitting paths are **independent mechanisms** that can both be enabled simultaneously (the default configuration has both active). Their interaction:

| Step | What fires | Condition |
|---|---|---|
| 1. Network pre-forward | Proactive split | `requestRange > effectiveAutoSplitThreshold` |
| 2. Upstream receives request | (sub-requests from step 1, or original if not proactively split) | — |
| 3. Network post-forward | Reactive split | Upstream returned `ErrCodeEndpointRequestTooLarge` or `-32012` |

**Key interaction rules:**

- **Proactive first, reactive never for those sub-requests.** If proactive splitting fires (step 1), it sets `CompositeType = CompositeTypeLogsSplitProactive` and returns the merged response; the request never reaches step 3 for the parent. The sub-requests generated by proactive splitting have `ParentRequestId` set, so they are **excluded from reactive splitting** by the re-entrancy guard at `architecture/evm/eth_getLogs.go:L373-L374`. This means if a proactively-generated sub-request itself gets a too-large error, that error propagates up and fails the entire parent — it cannot be reactively re-split.
- **Reactive fires only when proactive did not.** The post-forward hook only sees the original (unsplit) request when the effective proactive threshold was not exceeded (or no upstreams had a positive threshold). In that case, if the upstream rejects the single request as too large, reactive splitting engages.
- **Both enabled but different thresholds:** If `getLogsAutoSplittingRangeThreshold = 5000` (default) and the upstream only starts rejecting at ranges above 10,000 blocks, proactive splitting fires first (at 5001 blocks) and reactive never engages. If the upstream rejects ranges above 2,000 blocks but the threshold is 5,000, the 5,001-block request passes proactive (wait — 5001 > 5000 so proactive fires). The reactive path is primarily a backstop for requests that slipped through the proactive threshold or for upstreams with tighter limits than expected.
- **Chunk size contrast:** Proactive produces chunks of exactly `effectiveThreshold` blocks (last chunk may be smaller). Reactive always **bisects** — no relationship to `getLogsAutoSplittingRangeThreshold`. A reactive split of a 1,000-block range produces `[500, 500]` chunks regardless of configured thresholds.

Source: `architecture/evm/eth_getLogs.go:L141-L143` (proactive re-entrancy guard), `L373-L374` (reactive re-entrancy guard), `L240-L268` (proactive path), `L364-L412` (reactive path).

### Concurrent sub-request execution

`executeGetLogsSubRequests` (`architecture/evm/eth_getLogs.go:L650-L797`):
- Allocates `responses[len(subRequests)]` indexed by position (preserves order regardless of goroutine completion order).
- Limits concurrent in-flight sub-requests via a buffered semaphore channel of size `GetLogsSplitConcurrency` (default 10). (`architecture/evm/eth_getLogs.go:L662-L666`)
- Each sub-request: builds a new `JsonRpcRequest` via `BuildGetLogsRequest`, clones the parent's directives (carrying `SkipCacheRead` through), sets `ParentRequestId` to the parent's ID, copies the HTTP context (user headers, etc.), then calls `n.Forward(ctx, sbnrq)`. Sub-requests independently hit the cache, upstreams, and all the normal routing/retry/failsafe machinery.
- On any failure (build error, Forward error, nil JSON-RPC response, JSON-RPC error field set, clone error): increments `MetricNetworkEvmGetLogsSplitFailure` and appends the error. **If any sub-request fails, the entire parent request fails** — `errors.Join` is called and returned. (`architecture/evm/eth_getLogs.go:L783-L787`)
- `fromCache` for the merged result: `true` only if ALL sub-responses had `FromCache() == true`. (`architecture/evm/eth_getLogs.go:L796`)

### Merging and result ordering

`mergeEthGetLogsResults` wraps sub-responses in a `GetLogsMultiResponseWriter` (`architecture/evm/eth_getLogs.go:L799-L804`). This is a lazy writer: no concatenation happens until `WriteTo` is called (typically by the HTTP response writer). The writer:
- Skips `nil` and `IsResultEmptyish()` sub-responses when writing. (`architecture/evm/eth_getLogs.go:L442-L443`)
- Writes `[` then iterates responses in original index order, writing each sub-response's inner content (`WriteResultTo(..., trimSides=true)`) with `,` separators between non-empty chunks. Writes `]` at end. (`architecture/evm/eth_getLogs.go:L426-L473`)
- Result ordering is therefore deterministic and reflects proactive split order (ascending block range). No deduplication is performed — duplicate log entries across sub-requests (which should not occur for non-overlapping ranges) would be included.

`IsResultEmptyish()` returns `true` only when all sub-responses are nil or empty. (`architecture/evm/eth_getLogs.go:L476-L494`)

`Size()` sums sizes of all sub-responses. (`architecture/evm/eth_getLogs.go:L496-L509`)

`Release()` is idempotent; calls `Free()` on each sub-response and nils the slice. (`architecture/evm/eth_getLogs.go:L511-L528`)

### Empty result normalization (upstream post-forward)

If an upstream returns a successful response with an empty-ish result (`null`, `[]`, etc.), `upstreamPostForward_eth_getLogs` replaces it with a canonical `[]` array response while preserving metadata (cache status, block ref, duration, attempt counts). This ensures the merge layer always receives clean `[]` for blocks with no matching logs rather than `null`. (`architecture/evm/eth_getLogs.go:L349-L362`, `architecture/evm/common.go:L91-L115`)

### EIP-234 (blockHash) short-circuit

Requests using `blockHash` filter (EIP-234) are detected in all hooks and passed through without any splitting or range validation. (`architecture/evm/eth_getLogs.go:L101-L104`, `L162-L165`, `L312-L316`)

### Block availability enforcement (upstream pre-forward)

When `EnforceGetLogsBlockRange = true` (default), the upstream pre-forward hook calls `CheckBlockRangeAvailability` which:
1. Checks `toBlock` with `EvmAssertBlockAvailability(confidence=blockHead, forceRefresh=true)` — if not available, returns `ErrUpstreamBlockUnavailable` (retryable at upstream level). (`architecture/evm/block_range.go:L35-L57`)
2. Checks `fromBlock` with `forceRefresh=false` (historical, no need to re-fetch state). (`architecture/evm/block_range.go:L62-L81`)

### Future block empty result handling

When `eth_getLogs` returns empty for a block beyond the network's latest known head, `emptyResultBeyondConfidence` returns `true` (triggered in `upstreamPostForward_markUnexpectedEmpty` called for methods in `MarkEmptyAsErrorMethods`). However, `eth_getLogs` uses its own dedicated `upstreamPostForward_eth_getLogs` and calls `normalizeEmptyArrayResponse` unconditionally without invoking `upstreamPostForward_markUnexpectedEmpty`. So for `eth_getLogs`, empty is always normalized to `[]` regardless of whether the block is ahead of the chain tip — no retry is triggered for future-block empty results from `eth_getLogs`. (`architecture/evm/eth_getLogs.go:L349-L362`, `architecture/evm/hooks.go:L136-L139`)

### Behaviors & algorithms

#### Block tag resolution

`resolveBlockTagForGetLogs` (`architecture/evm/eth_getLogs.go:L25-L50`) is called for both `fromBlock` and `toBlock` values:
- Hex strings (`0x…`) are parsed directly.
- `latest` → `network.EvmHighestLatestBlockNumber(ctx)` (highest known block across all tracked upstreams).
- `finalized` → `network.EvmHighestFinalizedBlockNumber(ctx)`.
- `safe`, `pending`, `earliest`, or any unresolvable tag → returns `(0, 0)`, which causes all validation and splitting to be **skipped** — the request passes through to the upstream verbatim.

Source: `architecture/evm/json_rpc.go:L55-L83`

#### Proactive split sub-request generation

Given `fromBlock=F`, `toBlock=T`, `threshold=K`:
```
sub-requests: [F, F+K-1], [F+K, F+2K-1], …, [last_start, T]
```
where `last_start = F + floor((T-F)/K)*K` and the last chunk's `toBlock = T`.

Example: `F=1, T=5, K=2` → `[1,2], [3,4], [5,5]` (3 sub-requests). Verified by `TestNetworkPreForward_eth_getLogs/proactive_split_by_block_range_and_merge` (`architecture/evm/eth_getLogs_test.go:L1245-L1297`) and `TestHttp_EvmGetLogs_ProactiveRangeSplit_MergedResponse` (`erpc/http_server_logs_test.go:L168-L307`).

#### Effective threshold computation (multi-upstream)

The effective threshold = minimum positive `GetLogsAutoSplittingRangeThreshold` among all selected upstreams. Upstreams with `0` are excluded. Then capped at `GetLogsMaxAllowedRange` if set.

Example: upstream A has threshold=2, upstream B has threshold=4 → effective=2.
Verified by `TestNetworkPreForward_eth_getLogs/proactive_split_by_block_range_and_merge` using u1=2 and u2=4. (`architecture/evm/eth_getLogs_test.go:L1253-L1257`)

#### Reactive split priority (split-on-error)

Applied when upstream returns too-large error and block tags have been resolved to hex (split uses `extractBlockRange` which requires `0x` prefix):
1. `blockRange > 1`: bisect at `mid = fromBlock + (blockRange/2)` → `[from, mid-1]` and `[mid, to]`. Both halves inherit original `address` and `topics`.
2. `blockRange == 1` and `address` is an array with > 1 element: bisect address list at `len/2`.
3. `blockRange == 1` and single/no address and `topics[0]` is an `[]interface{}` with > 1 element: bisect `topics[0]` OR-list at `len/2`.
4. Otherwise: error "request cannot be split further" — original error is returned unchanged.

Odd-count splits: floor division for left half, ceiling for right. E.g., 3 blocks → `[from, from+1]` and `[from+2, to]` (left=2, right=1 but floor/ceil via integer ops gives left=floor(3/2)=1 from mid calc, so `[from, from]` left, `[from+1, to]` right — see test: range 0x1..0x5 = 5 blocks → `[1,2]` and `[3,5]`, i.e. `mid = 1+(5/2)=1+2=3`). (`architecture/evm/eth_getLogs_test.go:L262-L273`)

#### Hard-limit enforcement order and error shapes

Three hard limits are checked in sequence at network pre-forward (`architecture/evm/eth_getLogs.go:L207-L217`), all before any proactive splitting:

1. **Block range** (`GetLogsMaxAllowedRange`): checked first. If `requestRange > maxRange`, returns `ErrGetLogsExceededMaxAllowedRange` — HTTP 413, wire 200 (JSON-RPC error envelope), error code `ErrGetLogsExceededMaxAllowedRange`. Classified as `IsClientError = true`, `U:yes N:yes`. (`common/errors.go:L2666-L2692`)

2. **Addresses** (`GetLogsMaxAllowedAddresses`): checked second. If `addrCount > maxAddrs`, returns `ErrGetLogsExceededMaxAllowedAddresses` — HTTP 413, wire 200. Only triggered when `address` is a JSON array; a single-string address (or absent) is never counted. (`common/errors.go:L2693-L2697`, `architecture/evm/eth_getLogs.go:L171-L174`)

3. **Topics** (`GetLogsMaxAllowedTopics`): checked third. If `topicCount > maxTopics`, returns `ErrGetLogsExceededMaxAllowedTopics` — HTTP 413, wire 200, error code `ErrGetLogsExceededMaxAllowedTopics`. Classified as `IsClientError = true`, `U:yes N:yes`. (`common/errors.go:L2698-L2717`) Only `topics[0]` (the event-signature OR-list) is counted; topic positions 1–3 (AND conditions) are not counted. If `topics[0]` is a single string (non-array), count = 1; if it is an array, count = `len(topics[0])`. (`architecture/evm/eth_getLogs.go:L175-L182`)

All three errors are **not** `ErrCodeEndpointRequestTooLarge` and therefore **do not trigger `GetLogsSplitOnError` reactive splitting** — they are eRPC-originated client errors, not upstream-originated too-large errors. The client must reduce the query dimensions itself.

**`GetLogsMaxAllowedTopics` footgun:** the default is `0` (no limit). Setting it to, say, `10` means a client sending `topics: [["0xAAA", "0xBBB", ..., "0xKKK"]]` with 11 OR conditions will get HTTP 413 immediately, with no automatic splitting by topics OR-list. If the operator also wants eRPC to automatically split by topics when the upstream rejects, they must rely on `GetLogsSplitOnError = true` (not on `GetLogsMaxAllowedTopics`, which only caps client input before the request reaches any upstream).

#### Error triggering reactive split

Two error shapes trigger `GetLogsSplitOnError`:
1. `common.ErrCodeEndpointRequestTooLarge` — produced by eRPC when normalizing upstream HTTP 413 responses or upstream error messages that match known too-large patterns. This type has two complaint subtypes, both of which trigger splitting: `complaint=evm_block_range` (block-range too wide) and `complaint=evm_addresses` (too many addresses). (`common/errors.go:L2117`, `common/errors.go:L2121-L2122`)
2. `ErrJsonRpcExceptionInternal` with `NormalizedCode() == JsonRpcErrorEvmLargeRange` (`-32012`) — produced by RPC nodes that return `{"error":{"code":-32012,"message":"..."}}`. (`architecture/evm/eth_getLogs.go:L380-L390`)

The reactive split check at `architecture/evm/eth_getLogs.go:L379` uses `HasErrorCode(re, common.ErrCodeEndpointRequestTooLarge)` which matches any `ErrEndpointRequestTooLarge` regardless of the `complaint` field. Therefore **address-count-too-large errors** from vendors — normalized to `ErrEndpointRequestTooLarge` with `complaint=evm_addresses` — also trigger reactive splitting just as block-range-too-large errors do. The `complaint` field is informational metadata only; it does not gate the split.

**Vendor messages that normalize to `ErrEndpointRequestTooLarge` with `complaint=evm_addresses`:**
- Alchemy / DRPC: `"exceed max addresses or topics per search position"` (matches `strings.Contains(msg, "addresses or topics per search position")`)
- Infura: `"This query contains N filters. The current limit is 5000."` (matches `strings.Contains(msg, "filters") && strings.Contains(msg, "current limit is")`)
- Generic: `"please specify less number of address in the getLogs query"` (matches `strings.Contains(msg, "specify less number of address")`)

Source: `architecture/evm/error_normalizer.go:L102-L116`. Test: `TestExtractJsonRpcError_RequestTooLargeNormalization` covering all three cases at `architecture/evm/error_normalizer_test.go:L14-L57`.

**Split priority when address-count error fires on a multi-block request:** `splitEthGetLogsRequest` always tries block-range bisection first (`blockRange > 1` at `architecture/evm/eth_getLogs.go:L562-L578`). So if Alchemy rejects a 1,000-block request with 50 addresses due to the address+topic-count limit, eRPC will bisect the block range into two 500-block sub-requests — not split the address list. Only when the request already covers a single block does the split pivot to address list bisection. This means the address-count reactive path typically takes many recursion levels before reducing address count: it first halves the block range down to 1 block, then halves addresses.

Importantly: **eRPC's own `ErrGetLogsExceededMaxAllowedRange`** (code `ErrGetLogsExceededMaxAllowedRange`) does **not** trigger reactive split — it has its own error code and is checked separately at line L379 (`isTooLarge` check for `ErrCodeEndpointRequestTooLarge` would not match this). The network post-forward hook explicitly checks for the too-large codes and returns the original error if not matched. (`architecture/evm/eth_getLogs_test.go:L984-L994` — `TestNetworkPostForward_NoSplitOnNonTooLargeError`)

#### Cache interaction

Sub-requests flow through the full forward path including cache reads and writes. The merged parent response is marked `fromCache = true` only if all sub-requests were cache hits. If even one sub-request was a cache miss (upstream call), `fromCache = false`. (`architecture/evm/eth_getLogs.go:L796`)

The parent request's directive `SkipCacheRead` is propagated verbatim to all sub-requests. (`architecture/evm/eth_getLogs.go:L698`)

#### Empty-result behavior

A sub-response that is `nil` or `IsResultEmptyish()` (null or empty array) is silently skipped during the merge write — it does not contribute a `[]` or `,` separator to the output. This means if all sub-responses are empty, the merged output is `[]`. If some are empty and some are not, the non-empty logs are concatenated correctly without interleaved empties. Verified by `TestGetLogsMultiResponseWriter_WithEmptySubResponse` variants and `TestHttp_EvmGetLogs_SplitOnError_EmptyAndNonEmptyMergedSkipsEmpty`. (`architecture/evm/eth_getLogs_test.go:L780-L901`, `erpc/http_server_logs_test.go:L513-L597`)

`null` responses are treated same as `[]` (both are "emptyish"). (`architecture/evm/eth_getLogs_test.go:L869-L901`)

#### Deterministic output ordering

Sub-requests are dispatched concurrently but their results are stored in a positionally-indexed slice. The `GetLogsMultiResponseWriter` iterates this slice in insertion order. Even if `fromBlock=0x3` sub-request completes before `fromBlock=0x1`, the output always orders `0x1` logs before `0x2` before `0x3`. Verified by `TestExecuteGetLogsSubRequests_DeterministicOrder` which introduces artificial delays to scramble completion order. (`architecture/evm/eth_getLogs_test.go:L1011-L1050`)

### Edge cases & gotchas

1. **Any single sub-request failure aborts the entire split.** If 9 of 10 sub-requests succeed and 1 fails, `errors.Join` returns a multi-error and the caller receives an error response. The successful partial results are discarded. Source: `architecture/evm/eth_getLogs.go:L783-L787`. Test: `TestExecuteGetLogsSubRequests/partial_failure` (`architecture/evm/eth_getLogs_test.go:L399-L425`).

2. **`safe` and `pending` tags skip all validation and splitting.** These tags cannot be resolved to a numeric block by eRPC (only `latest` and `finalized` are supported). If either tag appears in `fromBlock` or `toBlock`, the request is forwarded verbatim — no range validation, no max-allowed-range check, no proactive splitting. The upstream must handle or reject them. (`architecture/evm/eth_getLogs.go:L192`, `architecture/evm/json_rpc.go:L77-L80`, `architecture/evm/eth_getLogs_test.go:L1385-L1405`)

3. **Proactive splitting takes priority over split-on-error.** The network pre-forward hook runs first; if it produces a split, the parent request is consumed and a merged response is returned before any upstream sees it. The network post-forward hook only fires if no pre-forward hook handled the request.

4. **`ErrGetLogsExceededMaxAllowedRange` does NOT trigger `GetLogsSplitOnError`.** If eRPC's own max-allowed-range hard limit fires, it returns HTTP 413 immediately. Only upstream-originated too-large errors trigger reactive splitting. (`architecture/evm/eth_getLogs.go:L379`, `architecture/evm/eth_getLogs_test.go:L984-L994`)

5. **Sub-requests inherit parent's `SkipCacheRead` directive but nothing else changes.** `UseUpstream` is not forced — the comment in code explicitly marks this as a TODO. Sub-requests therefore undergo full upstream selection, health checks, failsafe, and retry independently. (`architecture/evm/eth_getLogs.go:L696-L699`)

6. **Block range computation uses `0x`-prefixed hex only.** `extractBlockRange` (used by reactive split) requires `0x`-prefixed hex strings and returns an error if `fromBlock`/`toBlock` are not hex. Tag strings that failed resolution would have already been resolved to hex by the pre-forward hooks; tags like `pending` that were passed through (no resolution) would cause `splitEthGetLogsRequest` to fail — but in that scenario the request would never have had a `too-large` error checked anyway due to the tag-resolution short-circuit. (`architecture/evm/eth_getLogs.go:L629-L648`)

7. **`GetLogsMaxAllowedTopics` counts only `topics[0]` OR-list length.** Topics at positions 1, 2, 3 (AND conditions) are not counted toward the limit. If `topics[0]` is a single string (not an array), count = 1. Source: `architecture/evm/eth_getLogs.go:L175-L182`.

8. **`GetLogsMaxAllowedAddresses` counts only if `address` is `[]interface{}`.** A single-string address is not counted (count stays 0). Only when `address` is parsed as a JSON array does the count apply. Source: `architecture/evm/eth_getLogs.go:L171-L174`.

9. **Merged response `fromCache = false` if any sub-request was a cache miss.** This is conservative — even if 9/10 sub-requests hit cache, the merged response will not be marked from cache. (`architecture/evm/eth_getLogs.go:L796`)

10. **`Release()` on `GetLogsMultiResponseWriter` is idempotent** but calling `Size()` after `Release()` returns 0 (not an error). Tests verify this: `TestGetLogsMultiResponseWriter_ReleaseIdempotentSizeAfterRelease`. (`architecture/evm/eth_getLogs_test.go:L903-L915`)

11. **Concurrent write + Release is safe.** `GetLogsMultiResponseWriter` holds a `sync.RWMutex`; `WriteTo` acquires read lock, `Release` acquires write lock. The concurrent race test `TestGetLogsMultiResponseWriter_ConcurrentWriteAndFree_NoParseBodyPanic` exercises this path. (`architecture/evm/eth_getLogs_test.go:L919-L958`)

12. **`blockHash` filter (EIP-234) always bypasses all splitting logic.** Any filter with a non-empty `blockHash` key is passed directly to the upstream at all hook stages. (`architecture/evm/eth_getLogs.go:L101-L104`, `L162-L165`, `L312-L316`)

13. **Empty result from `eth_getLogs` is normalized to `[]`, not `null`.** The upstream post-forward hook replaces null/empty responses with a canonical `[]` JSON-RPC response. This applies even for future blocks (beyond chain tip) — eRPC does not retry empty `eth_getLogs` results, it normalizes them. Source: `architecture/evm/eth_getLogs.go:L357-L359`, `architecture/evm/common.go:L91-L115`.

14. **`fromBlock > toBlock` is rejected immediately at network pre-forward.** The error is `ErrInvalidRequest` (HTTP 400). This check runs even before hard limit checks, but only when both tags can be resolved to numeric blocks. (`architecture/evm/eth_getLogs.go:L196-L200`, `architecture/evm/eth_getLogs_test.go:L1176-L1187`)

15. **Re-entrancy for nested splits is prevented by `ParentRequestId` check, not `CompositeType`.** A sub-request from a proactive split has `ParentRequestId` set; it will not be re-split proactively. But if that sub-request then gets a too-large error at the upstream level, `networkPostForward_eth_getLogs` also checks `rq.ParentRequestId() != nil` and skips split-on-error too. So sub-requests from proactive splits cannot trigger reactive splits. (`architecture/evm/eth_getLogs.go:L141-L143`, `L373-L374`)

16. **Reactive splitting always bisects — there is no chunk-size config.** Unlike proactive splitting (which produces chunks of exactly `getLogsAutoSplittingRangeThreshold` blocks), reactive splitting always produces exactly 2 sub-requests via `mid = fb + blockRange/2`. A 1,000-block request that triggers a too-large error is split into `[500, 500]` blocks, not into `threshold`-sized chunks. If the upstream's limit is 100 blocks and `getLogsAutoSplittingRangeThreshold = 500`, reactive splitting will recursively bisect: 1000 → 500 → 250 → 125 → 62 → 31 → 15 → 7 → 3 → 1 — potentially many round-trips. Source: `architecture/evm/eth_getLogs.go:L562-L577`.

17. **`getLogsSplitConcurrency` is a per-operation limit, not a global limit.** N concurrent `eth_getLogs` requests each performing a split can each hold up to `getLogsSplitConcurrency` in-flight sub-requests simultaneously. Total potential in-flight sub-requests = `N * getLogsSplitConcurrency`. On providers with per-second rate limits, a brief spike in concurrent split requests can cause many 429 errors simultaneously. 429 errors are NOT `ErrCodeEndpointRequestTooLarge` and will not trigger further splitting — they will fail the parent request. Source: `architecture/evm/eth_getLogs.go:L660-L665`.

18. **`ErrGetLogsExceededMaxAllowedTopics` does not trigger reactive splitting.** It is eRPC's own client-error code (not `ErrCodeEndpointRequestTooLarge`), so `isTooLarge` check at `architecture/evm/eth_getLogs.go:L379` evaluates to false. The client gets HTTP 413 and must reduce `topics[0]` length. eRPC will not automatically split by topic OR-list on this error; only upstream-originated `-32012` or 413 errors trigger the `topics0` bisection path. Source: `architecture/evm/eth_getLogs.go:L379-L391`, `common/errors.go:L2698-L2717`.

19. **`GetLogsMaxAllowedTopics` and reactive topic splitting are orthogonal.** `GetLogsMaxAllowedTopics` is a hard client-side cap that rejects the request before it reaches any upstream. Reactive topic splitting (`topics0` dimension) is triggered by an upstream-originated too-large error and bisects `topics[0]` to avoid the upstream limit. An operator can set `GetLogsMaxAllowedTopics = 0` (no client cap) and rely entirely on `GetLogsSplitOnError = true` for transparent upstream topic splitting — or set a client cap to fail fast at a known limit. These settings do not interact at runtime. Source: `architecture/evm/eth_getLogs.go:L215-L217` (cap check) vs `L599-L623` (reactive topic bisection).

20. **Alchemy/DRPC address-count and Infura filter-count errors also trigger reactive splitting via `ErrEndpointRequestTooLarge`.** The error normalizer at `architecture/evm/error_normalizer.go:L102-L116` maps three vendor message patterns to `ErrEndpointRequestTooLarge` with `complaint=evm_addresses`: (a) Alchemy/DRPC `"exceed max addresses or topics per search position"`, (b) Infura `"This query contains N filters. The current limit is 5000."`, and (c) generic `"specify less number of address"`. The reactive split at `eth_getLogs.go:L379` checks only for `ErrCodeEndpointRequestTooLarge` (regardless of complaint), so all three patterns enable reactive splitting. Crucially, `splitEthGetLogsRequest` tries block-range bisection before address bisection — so on a multi-block request, eRPC will first recursively halve the block range down to a single block before it ever splits the address list. Test coverage: `architecture/evm/error_normalizer_test.go:L14-L57` (`TestExtractJsonRpcError_RequestTooLargeNormalization`).
