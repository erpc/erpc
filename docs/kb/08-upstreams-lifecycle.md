## L1 — Capability (CTO view)

An "upstream" is one concrete RPC endpoint (Alchemy key, self-hosted geth, gRPC BDS feed, …) that eRPC can route a request to. The upstream subsystem owns the full lifecycle of each endpoint: config normalization (including project-wide `upstreamDefaults` inheritance), background bootstrapping with automatic retry (chainId detection/validation, EVM state-poller startup), per-request gating (ignored/allowed methods, shadow mode, syncing state, `use-upstream` selector), block-data availability checks (full vs archive node windows), and operator-driven cordoning (manual removal from routing). The outcome it buys: endpoints can be added declaratively and eRPC figures out what each one can actually serve — wrong-chain endpoints are rejected, pruned full nodes never get archive queries, broken providers self-heal back into rotation, and operators get a kill switch (`cordon`) that survives metric windows.

## L2 — Mechanics (staff-engineer view)

**Object graph.** `upstream.Upstream` wraps: the parsed `*common.UpstreamConfig`, a transport client (`clients.ClientInterface`), an optional `common.Vendor` (settings generator + error normalizer), a per-method support cache (`supportedMethods sync.Map`), the shared health tracker, a list of per-(method, finality) failsafe executors, an optional rate-limiter auto-tuner, and (for EVM) an `EvmStatePoller` (upstream/upstream.go:L113-134). The minimal cross-package contract is the `common.Upstream` interface (Id/VendorName/NetworkId/Config/Forward/Cordon/Uncordon/IgnoreMethod/ShouldHandleMethod — common/upstream.go:L40-60).

**Config-time pipeline.** During `ProjectConfig.SetDefaults` each static upstream first inherits from `upstreamDefaults` via `ApplyDefaults` (only-if-unset semantics; Tags/Failsafe/Routing are all-or-nothing, EVM sub-fields merge per-field), then gets its own `SetDefaults` (id derivation, type=evm, allowMethods⇒ignoreMethods=`["*"]`, …) (common/defaults.go:L1152-1163, L1491-1594, L1596-1715). Upstreams whose endpoint scheme is NOT a native protocol (`http(s)://`, `ws(s)://`, `grpc://`, `grpc+bds://` — util/schemes.go:L5-12) are *converted into providers*: e.g. `endpoint: alchemy://KEY` is removed from `p.Upstreams` and becomes a `ProviderConfig{Vendor:"alchemy", Settings:{apiKey:KEY}}` with the rest of the upstream block carried as a `"*"` override (common/defaults.go:L1164-1171, L1193-1241, scheme→settings table L1243-1425).

**Runtime bootstrap.** `UpstreamsRegistry.Bootstrap` fire-and-forgets registration of all static upstreams so service startup never blocks (upstream/registry.go:L95-104). Each upstream becomes a named `util.BootstrapTask` (`upstream/<id>`, or `network/evm:<chainId>/upstream/<id>` when chainId is preconfigured) executed by a `util.Initializer` with auto-retry: task timeout 120s, exponential backoff 3s→130s, factor 1.5, retries forever unless the task returns a `TaskFatal` error (upstream/registry.go:L420-475; util/initializer.go:L175-192, L296-311, L558-611). The task: deep-copies the config (race-protection because `detectFeatures` mutates it), reuses an already-registered `*Upstream` with the same Id if present, calls `NewUpstream` then `Upstream.Bootstrap`, registers the result into the per-network maps, and asynchronously invokes `onUpstreamRegistered` (wired by the project to push the network's config into the upstream via `SetNetworkConfig` — erpc/projects_registry.go:L161-186).

**NewUpstream** builds one `upstreamExecutor` per failsafe entry plus a catch-all no-op executor so every (method, finality) resolves; wires the breaker-transition metric hook; looks up the vendor (by explicit `vendorName`, else by `OwnsUpstream` URL sniffing — thirdparty/vendors_registry.go:L54-71); if a vendor matches, lets it `GenerateConfigs` and **replaces the config with the first generated one**; fills `VendorName` (falling back to `guessVendorName()` ⇒ `"unknown-<root-domain>"`); creates/reuses the transport client keyed by `UniqueUpstreamKey` (upstream/upstream.go:L136-222). Client type comes from the endpoint scheme: `http/https` → JSON-RPC HTTP client; `ws/wss` → hard error "websocket client not implemented yet"; `grpc/grpc+bds` → gRPC BDS client; anything else → "unsupported endpoint scheme" (clients/registry.go:L84-119).

**Bootstrap (the method)** runs `detectFeatures`: requires `Type` set; for EVM, makes a real `eth_chainId` call, parses it, and (a) fails retryably on transport errors (provider may recover), (b) fails **fatally** (stops initializer retries) on a non-numeric chainId or a mismatch against the configured `evm.chainId`; on success stores the detected chainId, the derived networkId (`evm:<chainId>`), and flips `chainIdValidated` (upstream/upstream.go:L224-244, L1407-1469). It then creates + bootstraps the EVM state poller; a poller bootstrap error is logged but does NOT fail registration ("will retry in background") (upstream/upstream.go:L234-241).

**Network readiness.** `PrepareUpstreamsForNetwork(ctx, networkId)` (called on lazy network init) schedules provider-based upstream-generation tasks once per network (`network/<id>/provider/<providerId>`), then polls every 200ms until ≥1 upstream is registered for the network, with a 30s wait cap (clamped to the caller's deadline). On timeout/terminal states it distinguishes `ErrNetworkInitializing` (work still ongoing) from retryable `ErrNetworkNotSupported` (all providers terminal, still zero upstreams), with a 100ms grace re-check for in-flight registrations (upstream/registry.go:L155-293, task-state summarizer L529-562).

**Request path.** `Network.doForward` → `Upstream.Forward(ctx, req, byPassMethodExclusion, isHedgeAttempt)`. Unless bypassed, `shouldSkip` runs in order: (1) shadow upstreams always skip real traffic, (2) `evm.skipWhenSyncing` + poller says syncing, (3) `ShouldHandleMethod` (ignore-then-allow with wildcards, cached per method), (4) `use-upstream` directive matched via `UpstreamMatchesSelector` (id first, then tags for purely-positive patterns) (upstream/upstream.go:L1505-1548; common/matcher.go:L73-112). Skips surface as `ErrUpstreamRequestSkipped` wrapping the reason (upstream/upstream.go:L444-449). Then the upstream-level rate-limit budget is checked, and the call runs through the matching failsafe executor (4-tier method+finality > method > finality > catch-all priority — upstream/upstream.go:L372-408). Block-availability gating deliberately does NOT happen in `shouldSkip`: it is centralized at the network layer (`checkUpstreamBlockAvailability`) so a "block slightly ahead" can be classified retryable instead of being short-circuited by a non-retryable skip (upstream/upstream.go:L1537-1546; erpc/networks.go:L1905-1987).

**Block availability.** `EvmAssertBlockAvailability(method, confidence, forceFreshIfStale, blockNumber)` first enforces configured `blockAvailability` bounds (lower/upper, each one of `exactBlock` / `latestBlockMinus` / `earliestBlockPlus`, fail-open if computed min>max), then per confidence: `finalized` ⇒ block must be ≤ finalized block (optionally re-polling) AND within the legacy `maxAvailableRecentBlocks` window; `blockHead` ⇒ block must be ≤ latest (force-polling once if ahead and `forceFreshIfStale`) AND within the window. The lower-bound check re-polls latest only when the block is within 10 blocks of the pruning boundary (upstream/upstream.go:L965-1102, L1262-1306, bounds resolution L1106-1212). The network layer turns "not available" into `ErrUpstreamBlockUnavailable`, retryable iff the block is ahead of latest by ≤ `maxRetryableBlockDistance` (default 128), and triggers an async fresh poll so retries see updated heads; availability-check *errors* fail open (erpc/networks.go:L1919-1987, L1835-1881).

**Cordoning.** Cordon state lives on the health tracker's `(upstream, method, finality=All)` cell as an explicit `atomic.Bool` + reason — it is NOT a metric, survives rolling-window rotation, and is excluded from idle-sweep eviction (health/tracker.go:L104-117, L207-224, L598-615). `Cordon` records the OFF→ON timestamp and an event counter; `Uncordon` observes the cordon-duration histogram on ON→OFF (health/tracker.go:L793-849). Reads check the `"*"` wildcard scope first, then the specific method (health/tracker.go:L853-890). Producers: admin JSON-RPC methods `erpc_cordonUpstream` / `erpc_uncordonUpstream` / `erpc_listCordoned` (erpc/admin.go:L54-58, L590-729). Consumer: the selection-policy engine exposes `cordonedReason` to the JS policy whose default `removeCordoned()` step drops the upstream from routing (internal/policy/eval.go:L98-100, L254-256; erpc/admin.go:L590-604).

**Self-healing method support.** When an upstream responds "method not supported" (`ErrCodeEndpointUnsupported`), the network calls `u.IgnoreMethod(method)` in a goroutine; if `autoIgnoreUnsupportedMethods` is true the method is appended to `IgnoreMethods` under a write lock and the support cache is set to false — permanent for the process lifetime (erpc/networks.go:L1156-1159; upstream/upstream.go:L1308-1319).

**Shadow upstreams.** `shadow.enabled: true` upstreams are registered into a separate `networkShadowUpstreams` map (never in normal routing) (upstream/registry.go:L574-596). After a real response, the project mirrors the request to each shadow upstream (subject to `ShouldHandleMethod` and `sampleRate`, default 1.0), forwarding with `byPassMethodExclusion=true` so the shadow-skip check in `shouldSkip` doesn't block it; results are compared for misbehavior tracking (erpc/shadow.go:L16-130).

### Behaviors & algorithms

**Bootstrap state machine (per upstream).**
1. `UpstreamsRegistry.Bootstrap` spawns `registerUpstreams(appCtx, upsCfg…)` in a goroutine; one `BootstrapTask` per config; names: `upstream/<id>` or `network/evm:<chainId>/upstream/<id>` when `evm.chainId > 0` (upstream/registry.go:L95-104, L411-427).
2. Initializer dedupes tasks by name (`LoadOrStore`); a task already succeeded never re-runs; Pending/Failed/TimedOut tasks are CAS'd to Running and re-attempted (util/initializer.go:L196-215, L253-341).
3. Auto-retry loop: first delay 3s, multiply 1.5×, cap 130s, runs until `StateReady` or `StateFatal`; task fn timeout 120s (util/initializer.go:L175-192, L558-611). Fatal detection: any error in the chain implementing `IsTaskFatal() bool` flips the task to `TaskFatal` and stops retries (util/initializer.go:L296-311).
4. Task body: deep `cfg.Copy()` (race-protection vs `detectFeatures` mutation); reuse existing `*Upstream` with same Id from `allUpstreams`; else `NewUpstream`; `ups.Bootstrap(ctx)`; `doRegisterBootstrappedUpstream`; async `onUpstreamRegistered` (upstream/registry.go:L420-475).
5. `doRegisterBootstrappedUpstream` appends to `allUpstreams` unconditionally, dedupes by Id into `networkUpstreams[networkId]` or `networkShadowUpstreams[networkId]` (when `shadow.enabled`), and refreshes the lock-free `networkUpstreamsAtomic` snapshot (non-shadow only) (upstream/registry.go:L564-609).
6. `onUpstreamRegistered` (project wiring) requires `ups.NetworkId() != ""`, lazy-creates the Network, then `ups.SetNetworkConfig(ntw.cfg)` — which propagates network alias to `networkLabel` and config to the state poller (erpc/projects_registry.go:L161-186; upstream/upstream.go:L346-370).

**`NewUpstream` sequence** (upstream/upstream.go:L136-222): build executors (one per failsafe entry; error propagates; breaker hook wires `MetricUpstreamBreakerStateChange`) → append catch-all no-op executor → vendor lookup → construct struct, `networkLabel="n/a"` → `initRateLimitAutoTuner` → if vendor: `GenerateConfigs` and **replace config with `cfgs[0]`** (error if zero configs) → fill `VendorName` (vendor name or guess) → `GetOrCreateClient` keyed by `UniqueUpstreamKey` → log `prepared upstream`.

**`detectFeatures` / chainId** (upstream/upstream.go:L1407-1469): error if `Type` empty; EVM: ensure `cfg.Evm` non-nil; `EvmGetChainId` issues literal `{"jsonrpc":"2.0","id":75412,"method":"eth_chainId","params":[]}` through `Forward(…, byPassMethodExclusion=true, …)`, normalizes hex → decimal string (upstream/upstream.go:L874-910). Failure modes: RPC error ⇒ retryable `ErrUpstreamClientInitialization{ErrUpstreamChainIdDetectionFailed}`; non-numeric ⇒ `NewTaskFatal(...)`; configured-vs-detected mismatch ⇒ `NewTaskFatal(ErrUpstreamChainIdMismatch)`. Success: `cfg.Evm.ChainId = detected`, `networkId = evm:<chainId>`, `chainIdValidated = true`.

**`PrepareUpstreamsForNetwork` wait algorithm** (upstream/registry.go:L155-293): per-network mutex; provider tasks scheduled once per network via `sync.Once` (`network/<id>/provider/<pid>`; provider task: `SupportsNetwork` → `GenerateUpstreamConfigs` → `registerUpstreams`) (upstream/registry.go:L162-186, L477-510). Wait: `minReady=1`; `waitTimeout=30s` clamped to caller deadline (min 50ms); 200ms ticker. Each tick: if count ≥ 1 ⇒ success; else consult `summarizeNetworkTasks` over prefixes `network/<id>/provider/`, `network/<id>/upstream/`, `upstream/` — ongoing work ⇒ keep waiting; all providers terminal ⇒ 100ms grace re-check then retryable `ErrNetworkNotSupported`. On timeout: registered >0 ⇒ success; ongoing ⇒ `ErrNetworkInitializing`; providers terminal ⇒ grace ⇒ `ErrNetworkNotSupported`; default `ErrNetworkInitializing` (upstream/registry.go:L188-292, L529-562).

**Upstream selection surface.** `GetSortedUpstreams(ctx, networkId, method)` is currently a pass-through over `GetNetworkUpstreams` (raw registration order; `method` ignored; returns `ErrNoUpstreamsFound` when empty) — the policy engine will own ordering in "Phase 7" (upstream/registry.go:L380-395). `GetNetworkUpstreams` reads the lock-free atomic snapshot, falling back to RLock+copy+populate (upstream/registry.go:L345-372). `RefreshUpstreamNetworkMethodScores` is a retained no-op (upstream/registry.go:L405-409). `OverrideOrderForTest` pins order for tests only (upstream/registry.go:L301-343).

**`shouldSkip` order & errors** (upstream/upstream.go:L1505-1548):
1. `shadow.enabled` ⇒ `ErrUpstreamShadowing` (HTTP n/a; common/errors.go:L1416-1430).
2. `evm.skipWhenSyncing == true` AND `EvmSyncingState() == Syncing` ⇒ `ErrUpstreamSyncing` (HTTP 422; common/errors.go:L1396-1414).
3. `!ShouldHandleMethod(method)` ⇒ `ErrUpstreamMethodIgnored` (HTTP 406; common/errors.go:L1375-1394).
4. `directives.use-upstream` set and `UpstreamMatchesSelector` false ⇒ `ErrUpstreamNotAllowed` (common/errors.go:L1432-1445).
`Forward` wraps any skip reason as `ErrUpstreamRequestSkipped` (HTTP 406; common/errors.go:L1328-1347; upstream/upstream.go:L444-449). Outcome classification maps both `ErrUpstreamRequestSkipped` and `ErrUpstreamMethodIgnored` to `skipped` (upstream/upstream.go:L49-51).

**`ShouldHandleMethod` algorithm** (upstream/upstream.go:L1361-1405): consult `supportedMethods` cache; else default `v=true`; scan `IgnoreMethods` — first wildcard match sets `v=false`; then scan `AllowMethods` — first match sets `v=true` (allow overrides ignore); cache and return. Pattern errors propagate as `(false, err)`. Cache means later config mutations only affect not-yet-cached methods (`IgnoreMethod` explicitly stores `false` to handle its own case — upstream/upstream.go:L1318).

**`use-upstream` selector matching** (common/matcher.go:L73-112): evaluate pattern against upstream id first (full boolean/glob grammar); on no match, and only for patterns WITHOUT `!`, evaluate against each tag. Negated patterns never match by tag (prevents a tag from "rescuing" an excluded upstream). Empty pattern matches nothing.

**`Forward` hot path** (upstream/upstream.go:L410-871): method parse errors → `ErrUpstreamRequest`; skip check (unless bypassed); resolve rate-limit budget (`GetBudget`, `GetRulesByMethod`, `TryAcquirePermit` with scope "upstream") → `ErrUpstreamRateLimitRuleExceeded` on deny; `nrq.SetLastUpstream(u)`; client-type switch (HttpJsonRpc and GrpcBds share the path; other types → `ErrUpstreamClientInitialization` "unsupported client type"). `tryForward` (one attempt):
- defer records `common.UpstreamAttempt{UpstreamId, VendorName, Outcome, Reason, IsHedge, IsRetry, AttemptIdx, ErrorCode (fingerprint), ErrorDetail (truncated to `AttemptErrorDetailMaxLen` = 200)}` into the request `ExecState`, plus `MetricUpstreamAttemptOutcomeTotal` and `MetricUpstreamSelectionTotal` increments (upstream/upstream.go:L509-569, L101-111).
- non-hedge attempts call `metricsTracker.RecordUpstreamRequest` (hedges excluded from rate counters by design — comment L575-595) and increment `MetricUpstreamRequestTotal`; latency timer started.
- `Client.SendRequest`; success (JSON-RPC result, no error field) ⇒ `SetLastValidResponse`, observe `MetricUpstreamResponseSizeBytes` for non-empty results.
- error paths: skipped ⇒ `MetricUpstreamSkippedTotal`; missing data ⇒ `MetricUpstreamMissingDataErrorTotal`; cancelled ⇒ neither failure tracking nor latency observation; capacity-exceeded ⇒ `recordRemoteRateLimit` (tracker + auto-tuner); all other errors ⇒ `RecordUpstreamFailure` (non-hedge) + `MetricUpstreamErrorTotal` with severity + fingerprint. Latency quantile only observed for success-non-emptyish and for EVM reverts (`ErrCodeEndpointExecutionException`) — cancellations, other errors, and emptyish responses are excluded (upstream/upstream.go:L676-805). Errors wrapped as `ErrUpstreamRequest` with attempt/retry/hedge counts.
- Executor wraps `tryForward` with retry/hedge/breaker/timeout; bare `ErrDynamicTimeoutExceeded` is wrapped as `ErrFailsafeTimeoutExceeded(ScopeUpstream)`; timeout-attribution emits `MetricNetworkTimeoutFiredTotal{scope="upstream"}` when this scope's timeout fired and retry-exhausted didn't win (upstream/upstream.go:L814-860).

**`classifyUpstreamOutcome` mapping** (upstream/upstream.go:L36-78), in order: request-canceled/`context.Canceled` ⇒ `cancelled`; failsafe-timeout/`ErrDynamicTimeoutExceeded` ⇒ `timeout`; breaker-open ⇒ `breaker_open`; skipped/method-ignored ⇒ `skipped`; capacity/any rate-limit code ⇒ `rate_limited`; missing-data ⇒ `missing_data`; execution-exception ⇒ `exec_revert`; block-unavailable ⇒ `block_unavailable`; transport ⇒ `transport_error`; server-side ⇒ `server_error`; client-side ⇒ `client_error`; unknown error ⇒ `server_error`; nil error + emptyish ⇒ `empty`; else `success` (enum values common/exec_state.go:L18-30). Selection reasons: `primary` / `retry` (snap.Retries>0) / `hedge` (common/exec_state.go:L40-42; upstream/upstream.go:L520-525).

**Failsafe executor matching** (upstream/upstream.go:L372-408): 4 passes over `failsafeExecutors` — (1) method≠`*` AND finalities non-empty: wildcard-match method AND finality∈list; (2) method≠`*`, no finalities; (3) method=`*` with finality match; (4) catch-all `*`. First hit wins; list order preserves config order within a tier.

**`upstreamExecutor` semantics** (upstream/upstream_executor.go): nil cfg ⇒ no-op executor with method `"*"` (L38-44); consensus rejected (L46-53); `emptyResultAccept` defaults to `DefaultEmptyResultAccept()` = [eth_getLogs, trace_filter, arbtrace_filter, eth_call, eth_getBalance, eth_getCode, eth_getStorageAt, eth_getTransactionCount] (L70-74; common/defaults.go:L2013-2026). `Run`: internal requests (`req.IsInternal()`) bypass retry+hedge+breaker, keep per-attempt timeout (L109-130). Retry loop: maxAttempts = `retry.maxAttempts` (≥1); increments ExecState UpstreamAttempts/UpstreamRetries; `shouldRetry` = false for nil error, composite requests, non-retryable EVM writes, execution-exceptions (unless `retryableTowardNetwork` detail true), missing-data with explicit `retryEmpty=false` directive; else `IsRetryableTowardsUpstream(err)` (L172-258). Hedge: skipped for composite/write methods; hedge attempts tagged via context key; `keep` rejects retryable-toward-network errors; ExecState UpstreamHedges incremented per fire (L267-343). Breaker: hedges, internal, and composite requests neither consume permits nor record outcomes (L345-385); outcome classification: cancel/skip ⇒ ignore; 5xx/transport/unauthorized/billing ⇒ failure; success-but-syncing-and-emptyish ⇒ failure; success ⇒ success (L391-421).

**Block availability — `EvmAssertBlockAvailability`** (upstream/upstream.go:L965-1102):
1. Guards: nil upstream/config; state poller present and non-null; `Type==evm` && `Evm != nil` else error.
2. Bounds gate: `resolveAvailabilityBounds()`; block < min ⇒ false + `MetricUpstreamStaleLowerBound` + debug log "block rejected: below lower availability bound"; block > max ⇒ false + `MetricUpstreamStaleUpperBound` (L982-1017).
3. `confidence == finalized` (enum value 2 — common/architecture_evm.go:L33-38): `EvmIsBlockFinalized` (≤ poller's finalized; if not and `forceFreshIfStale`, re-poll finalized once) — not finalized ⇒ false + upper-bound metric; then if `MaxAvailableRecentBlocks > 0`, `assertUpstreamLowerBound` must pass (L1019-1056).
4. `confidence == blockHead` (enum 1): block > latest AND `forceFreshIfStale` ⇒ one forced `PollLatestBlockNumber`; still > latest ⇒ false + upper-bound metric; then optional lower-bound check; final answer `blockNumber <= latestBlock` (L1057-1098).
5. Unknown confidence ⇒ error "unsupported block availability confidence" (L1099-1101).

**`assertUpstreamLowerBound`** (upstream/upstream.go:L1262-1306): `firstAvailable = latest − maxAvailableRecentBlocks`; immediate false + lower-bound metric if below. If within range AND within 10 blocks of the boundary (`|block − firstAvailable| ≤ 10`), force a fresh poll (guards pruning races on full nodes) and re-evaluate; poll failure or latest ≤ 0 ⇒ error.

**`resolveAvailabilityBounds`** (upstream/upstream.go:L1106-1212): nil Evm ⇒ (MinInt64, MaxInt64). No `blockAvailability` but `maxAvailableRecentBlocks > 0` and latest known ⇒ (latest − N, MaxInt64). Per bound: `exactBlock` ⇒ value; `earliestBlockPlus` ⇒ `EarliestBlock(probe)` + N (earliest 0 ⇒ 0+N; negative ⇒ unbounded); `latestBlockMinus` ⇒ latest − N when latest > 0, else unbounded; no kind set ⇒ unbounded. If both sides finite and min > max ⇒ WARN "computed block availability bounds are invalid (min > max); treating as unbounded for safety" ⇒ fully unbounded (fail-open).

**Effective tip helpers**: `EvmEffectiveLatestBlock` / `EvmEffectiveFinalizedBlock` return `min(pollerValue, upperBound)`; 0 for nil poller/null/value ≤ 0 (upstream/upstream.go:L1217-1249). `EvmBlockAvailabilityBounds` exposes the resolved pair for post-forward wrong-empty classification (L1256-1258).

**Network-level enforcement** (`checkUpstreamBlockAvailability`, erpc/networks.go:L1919-1987, invoked per-upstream pre-forward at L1278): only EVM architecture; gate on `resolveEnforceBlockAvailability(method, upstream)` whose precedence is (1) method-level user override differing from system default, (2) network `evm.enforceBlockAvailability`, (3) upstream has explicit bounds ⇒ true, (4) system default for method (`DefaultWithBlockCacheMethods`), (5) fallback true (erpc/networks.go:L1786-1831). Block number taken from cached normalization, else re-extracted; unknown (≤0) ⇒ fail-open. Calls `EvmAssertBlockAvailability(confidence=blockHead, forceFreshIfStale=true)`; check error ⇒ fail-open with debug log. Unavailable ⇒ `ErrUpstreamBlockUnavailable` (HTTP 503; details incl. blockNumber/latest/finalized — common/errors.go:L1349-1373), retryable iff `block > latest > 0` and `distance ≤ maxRetryableBlockDistance` (default 128). `handleBlockSkip` then logs, emits `MetricUpstreamErrorTotal` (severity info), marks the attempt; non-retryable cases are wrapped as `ErrUpstreamRequestSkipped`; retryable cases fire an async 10s-bounded `PollLatestBlockNumber` so the retry sees a fresh head (erpc/networks.go:L1835-1881).

**Cordoning algorithms** (health/tracker.go):
- `Cordon(up, method, reason)` (L793-818): store on `(upstream, method, DataFinalityStateAll)` cell; `Cordoned.Swap(true)`; reason stored; on OFF→ON only: `CordonedAtMs = now`, `MetricUpstreamCordonEventTotal{action="cordon"}`; always set `MetricUpstreamCordoned` gauge to 1 (gauge cached per (upstream, method, reason) — L457-466).
- `Uncordon(up, method, reason)` (L820-849): swap false, clear reason; on ON→OFF: observe `MetricUpstreamCordonDurationSeconds` with `now − CordonedAtMs`, emit event counter `action="uncordon"`; gauge set 0.
- `IsCordoned(up, method)` (L853-863): true if `(up, "*", All)` cordoned, else check `(up, method, All)`.
- `CordonedReason(up, method)` (L870-890): same wildcard-then-specific fallback, returns (reason, true) when cordoned. Surfaced through `Upstream.Cordon/Uncordon/CordonedReason` (upstream/upstream.go:L1568-1580).
- Persistence: `Rotate()` (rolling-window tick) never touches Cordoned (L207-224); idle sweep never evicts cordoned cells (L598-615); only `Reset()` (test/admin helper) clears it (L226-236).

**Cordon admin RPCs** (erpc/admin.go): dispatch on `erpc_cordonUpstream` / `erpc_uncordonUpstream` / `erpc_listCordoned` (L54-58). Params object: `{projectId, upstream, method?, reason?}`; projectId+upstream required; method defaults to `"*"` (wholesale cordon); reason defaults to "admin: manual cordon"/"admin: manual uncordon" (L606-636, L658-687). `erpc_listCordoned{projectId}` returns rows of `{upstream, reason}` for upstreams whose `"*"` scope is cordoned — method-scoped cordons are NOT listed (L692-729, check at L721).

**Identity & client reuse.** `UniqueUpstreamKey = <id>/<hex(sha256(id‖endpoint‖networkId‖headers…))>` — used as the client-registry cache key so an endpoint/network/header change yields a new client (common/upstream.go:L62-80; clients/registry.go:L47-53, L121-123).

**Config marshalling.** `MarshalJSON`/`MarshalYAML` on `UpstreamConfig` redact the endpoint: native protocols keep `scheme://host#redacted=<hash5>`; `*envio` keeps scheme+host; `*repository` keeps scheme+host+hash; others collapse to `scheme#redacted=<hash5>` (common/config.go:L1033-1049; util/redact.go:L10-36).

### Edge cases & gotchas

1. **Negative/zero block numbers pass blockHead availability.** `EvmAssertBlockAvailability(..., -1)` and `(..., 0)` return true (no lower bound configured; `bn <= latest`); the network-level gate also skips entirely when extracted block ≤ 0 (fail-open) (upstream/upstream_block_availability_test.go:L78-108; erpc/networks.go:L1940-1943).
2. **Unknown confidence value errors.** Confidence other than 1/2 returns `(false, "unsupported block availability confidence")` (upstream/upstream.go:L1099-1101; test upstream/upstream_block_availability_test.go:L110-125).
3. **Lower-bound re-poll race can reject an initially-valid block.** Within 10 blocks of the pruning boundary a fresh poll runs; if the chain advanced, `firstAvailable` moves up and the block flips to unavailable mid-check (documented intentionally in tests) (upstream/upstream.go:L1279-1305; upstream/upstream_block_availability_test.go:L723-806). Conversely blocks far inside the window never trigger a poll (test L562-591).
4. **`maxAvailableRecentBlocks: 0` = archive semantics.** Any block ≤ latest is available under blockHead confidence (upstream/upstream.go:L1087-1098; test upstream/upstream_block_availability_test.go:L291-312).
5. **Two distinct fail-opens.** (a) invalid computed bounds (finite min > max, e.g. `earliestBlockPlus` before earliest detection completes) ⇒ whole bounds disabled with WARN (upstream/upstream.go:L1199-1210); (b) any *error* during the network availability check ⇒ request proceeds (erpc/networks.go:L1949-1959).
6. **Force-poll only happens when block is ahead.** With `forceFreshIfStale=true`, blockHead confidence polls latest only if `blockNumber > cachedLatest`; an up-to-date cache means zero polls (upstream/upstream.go:L1061-1069; tests upstream/upstream_block_availability_test.go:L469-529). Poll errors propagate as `(false, err)` (test L127-156).
7. **`UpstreamConfig.Copy()` is only partially deep.** Tags slice header, `Shadow`, and `Routing` pointers are shared between original and copy (`*copied = *c` then only Evm/Failsafe/JsonRpc/Grpc/RateLimitAutoTune/IgnoreMethods/AllowMethods are deep-copied) (common/config.go:L942-980). The bootstrap task relies on `Copy()` for race-protection (upstream/registry.go:L421-422) — runtime mutation of shared sub-objects would still race.
8. **`JsonRpcUpstreamConfig.Copy()` does NOT copy the Headers map.** `*copied = *c` aliases the map and `maps.Copy(copied.Headers, c.Headers)` copies it onto itself; original and copy share one Headers map (contrast `GrpcUpstreamConfig.Copy` which allocates) (common/config.go:L1081-1094 vs L1104-1118).
9. **`UniqueUpstreamKey` is order-unstable with ≥2 JsonRpc headers.** The sha256 ingests headers via Go map iteration (`for k, v := range`), whose order is randomized — two calls can produce different keys, defeating the client cache lookup and re-creating the client (common/upstream.go:L65-80; clients/registry.go:L47-53).
10. **Vendor `GenerateConfigs` replaces the config and only `cfgs[0]` is used**; zero generated configs aborts upstream creation ("no upstreams generated by vendor") (upstream/upstream.go:L191-200).
11. **ChainId failure modes differ.** Transport/RPC failure during detection retries forever (backoff capped 130s — util/initializer.go:L558-611); non-numeric chainId or configured-vs-detected mismatch is `TaskFatal` and permanently stops retries for that upstream (upstream/upstream.go:L1419-1455; util/initializer.go:L296-311).
12. **State-poller bootstrap failure does not block registration.** The upstream still registers and serves; the poller retries in background — but availability checks needing latest-block data will error/fail-open until it recovers (upstream/upstream.go:L234-241).
13. **`allowMethods` without `ignoreMethods` silently becomes allow-list-only** (`ignoreMethods=["*"]` injected at defaults time) (common/defaults.go:L1706-1712).
14. **Method-support results are cached forever per process.** `ShouldHandleMethod` caches per method name in `supportedMethods`; later edits to Ignore/Allow lists don't invalidate existing entries (only `IgnoreMethod` writes its own `false` entry) (upstream/upstream.go:L1361-1405, L1318).
15. **`IgnoreMethod` is a silent no-op when `autoIgnoreUnsupportedMethods` is nil/false** — and there is NO system-wide default making it true; only the repository provider sets it (upstream/upstream.go:L1308-1313; thirdparty/repository.go:L89-91).
16. **Wildcard ignore patterns honored by tests**: `eth_*` matches `eth_getBalance` but not `net_version`; `eth_*_*` matches `eth_get_balance` but NOT `eth_getBalance` (single `*` does not cross the second underscore requirement) (upstream/upstream_test.go:L33-49, L251-271).
17. **Uncordoning a method does not clear a wildcard cordon.** `Uncordon(method)` flips only that cell; `IsCordoned`/`CordonedReason` check `"*"` first, so a `"*"`-scope cordon still wins (health/tracker.go:L820-849, L853-890). Also `erpc_listCordoned` only reports `"*"`-scope cordons (erpc/admin.go:L720-723).
18. **Repeated `Cordon` calls update the reason but not the start timestamp** (duration accounting survives reason edits); the gauge is per-(method, reason) label set, so changing reason creates a new gauge series while the old one stays at 1 until a matching set(0) — the event counter dedupes, the gauge does not (health/tracker.go:L805-817).
19. **Cordoned tracker cells are immune to idle eviction; wildcards too.** The sweep skips method `"*"` cells and any cordoned cell so admin state can't vanish after 30 idle minutes (health/tracker.go:L598-615, comments L581-590).
20. **`allUpstreams` may hold duplicate pointers.** Registration appends unconditionally; only the per-network maps dedupe by Id. If the same upstream Id is bootstrapped under two task names (e.g. once as `upstream/<id>` lazily and once as `network/<nid>/upstream/<id>`), the same `*Upstream` is re-registered and appended twice (upstream/registry.go:L436-452, L564-609).
21. **Stale comment hazard:** the provider-task-prefix comment says `provider/<id>/network/<networkId>` but actual names are `network/<networkId>/provider/<id>` — trust `buildProviderBootstrapTask` and `summarizeNetworkTasks` (upstream/registry.go:L512-514 vs L481, L530-531).
22. **`PrepareUpstreamsForNetwork` returns success on timeout if ≥1 upstream is ready** — partial readiness is acceptable by design (`minReady=1`); other upstreams keep bootstrapping in background (upstream/registry.go:L188-292).
23. **Hedge attempts are invisible to per-upstream rate counters and latency quantiles**; only EVM reverts among errors feed latency; emptyish successes are excluded from quantiles too — selection scoring sees a deliberately filtered signal (upstream/upstream.go:L575-595, L741-772, L802-804).
24. **`EvmGetChainId` bypasses method exclusion** (`byPassMethodExclusion=true`), so `ignoreMethods: ["*"]` cannot break bootstrap; but it still consumes the upstream rate-limit budget (upstream/upstream.go:L874-881, L457-493).
25. **Shadow mirroring also bypasses method exclusion at Forward level** but pre-checks `ShouldHandleMethod` manually, so ignoreMethods are honored while the shadow-skip flag is not (erpc/shadow.go:L53-62, L118-119).
26. **`ws://`/`wss://` endpoints pass config validation (native protocol) but fail at client creation** with "websocket client not implemented yet" — a retryable bootstrap error that will retry forever (clients/registry.go:L100-101; util/initializer.go:L558-611).
27. **Test-only ordering override exists in production code** (`OverrideOrderForTest`) because in-package helpers can't live in `_test.go` for cross-package use; it also refreshes the atomic snapshot (upstream/registry.go:L301-343).
28. **`CircuitBreakerState()` reports only the catch-all executor's breaker** — method-specific breakers are invisible to that diagnostic surface (upstream/upstream.go:L309-333).
29. **`EvmEffectiveLatestBlock` returns 0 (not latest) when poller value ≤ 0**, and ignores upper bounds that are ≥ latest — equal bound returns the raw latest (tests upstream/upstream_block_availability_test.go:L888-1027).
30. **`upstreamDefaults.failsafe` is pointer-shared into upstreams in `ApplyDefaults`** (`u.Failsafe = defaults.Failsafe`), then `SetDefaults` may deep-copy entries; mutation ordering matters if anything edits failsafe configs at runtime (common/defaults.go:L1511-1513 vs L1663-1672).
31. **`ErrUpstreamRequestSkipped` (code #27) is a PERMANENT skip; `ErrUpstreamBlockUnavailable` (code #28) is a TRANSIENT skip.** `ErrUpstreamRequestSkipped` wraps reasons like shadow mode, method ignored, `use-upstream` mismatch, or syncing state — the upstream cannot serve this request at all and will not be retried. `ErrUpstreamBlockUnavailable` means the upstream doesn't have the block *yet* but may soon (the block may be in flight); the network layer classifies it retryable iff `block > latest > 0` and `distance ≤ maxRetryableBlockDistance` (default 128). When non-retryable (block too far ahead or distance exceeded), the network layer wraps it inside `ErrUpstreamRequestSkipped` to stop retry attempts (erpc/networks.go:L1862-1881; common/errors.go:L1328-1374). HTTP 503 vs 406 status codes reflect this distinction (`ErrUpstreamBlockUnavailable.ErrorStatusCode()=503`; `ErrUpstreamRequestSkipped.ErrorStatusCode()=406`).
32. **`ErrUpstreamNodeTypeMismatch` (#60) is defined but not fired by any current code path.** The error type exists at common/errors.go:L2097-2113 with `expected` and `actual` `EvmNodeType` detail fields. Prior code in `nodeType`-specific logic produced it when an archive-only query was routed to a full node; that logic was removed when `blockAvailability` bounds replaced `nodeType` semantics (upstream/upstream.go:L1460 comment: "NodeType-specific logic removed"). As of the current codebase, no call site invokes `NewErrUpstreamNodeTypeMismatch`. The error code still appears in `ErrUpstreamsExhausted.SummarizeCauses()` as a recognized category (common/errors.go:L1015-1018) for backward compatibility.
33. **`ErrUpstreamNetworkNotDetected` (#25) is defined but not fired by any current code path.** The error type exists at common/errors.go:L1292-1311 with `projectId` and `upstreamId` detail fields and message "network not detected for upstream either from config nor by calling the endpoint". It was intended for cases where `detectFeatures` cannot determine the network (no configured chainId and no successful `eth_chainId` call). In the current code, `detectFeatures` either returns `ErrUpstreamClientInitialization` (transient RPC failure) or `TaskFatal(ErrUpstreamClientInitialization)` (non-numeric chainId or mismatch) — `ErrUpstreamNetworkNotDetected` is never constructed. Has no `ErrorStatusCode()` method (no HTTP status override; inherits default behavior). No `IsRetryableTowardsUpstream` override, so it defaults to retryable.
34. **`ErrUpstreamInitialization` (#26) is used in a narrow path only.** Produced only in `architecture/evm/block_range.go:L27` when `CheckBlockRangeAvailability` is called but the upstream's `EvmStatePoller` is nil or null. This is distinct from `ErrUpstreamClientInitialization` (code `ErrUpstreamClientInitialization`) which covers bootstrap-time chainId detection failures. `ErrUpstreamInitialization` means the upstream bootstrapped but its poller never started successfully; the error is retryable by default (no `IsTaskFatal`). Detail field: `upstreamId` (common/errors.go:L1318-1326).
35. **`ErrNoUpstreamsDefined` (#21) is defined but currently not called by any code path.** The error type exists at common/errors.go:L1217-1231 with HTTP 404 status code, meaning "project has zero upstreams configured". In practice, config validation at `ProjectConfig.Validate` enforces "upstreams or providers required" with a plain `fmt.Errorf` (not this typed error — common/validation.go:L607-609). The registry returns `ErrNoUpstreamsFound` (different error, HTTP 404) when a specific network has no upstreams registered (upstream/registry.go:L392). `ErrNoUpstreamsDefined` is the conceptual distinction: the project itself has no upstreams at all (vs `ErrNoUpstreamsFound` which means upstreams exist but none for a specific network).
36. **`ErrEndpointMissingData` (#59) carries three diagnostic detail fields** populated from the upstream's state at the time of the error: `latestBlock` (poller's current latest), `finalizedBlock` (poller's current finalized), `maxAvailableRecentBlocks` (from `cfg.Evm.MaxAvailableRecentBlocks`). These are populated in `NewErrEndpointMissingData` when the upstream implements `EvmUpstream` and has a non-nil state poller (common/errors.go:L2064-2090). The HTTP status is 200 (not an error from the client's perspective — common/errors.go:L2092-2095). These fields are accessible via `DeepSearch` and used downstream for logging and retry decisions.
37. **`TaskFatalError` (#74) is a control-flow signal, not a `StandardError`.** Unlike all other errors in the upstream lifecycle, `TaskFatalError` does NOT implement the `StandardError` interface — it has no `Base()`, no `HasCode()`, no `CodeChain()`. It wraps an underlying error and implements only `Error() string`, `Unwrap() error`, and `IsTaskFatal() bool`. The initializer checks `IsTaskFatal()` via interface assertion on every error returned by a bootstrap task (`util/initializer.go:L296-311`); if true, the task is marked `StateFatal` and never retried. Currently used in two places in upstream bootstrap: non-numeric chainId and chainId mismatch — both via `common.NewTaskFatal(common.NewErrUpstreamClientInitialization(...))` (upstream/upstream.go:L1437-1454). The constructor is `common.NewTaskFatal(err error) error` and the predicate is `e.IsTaskFatal() bool` (common/errors.go:L421-436).
38. **`upstreamDefaults` all-or-nothing tags inheritance surprises when mixing defaults and per-upstream tags.** If `upstreamDefaults.tags = [region:us, tier:primary]` and upstream A has `tags: [tier:backup]`, upstream A gets only `[tier:backup]` — it does NOT inherit `region:us`. To achieve partial override, the operator must specify ALL intended tags on each upstream explicitly. The condition is `len(u.Tags) == 0` — even a single tag blocks all inheritance (common/defaults.go:L1508-1510).
39. **`vendorName` set to a non-existent vendor name returns `nil` from `LookupByUpstream` silently.** When `vendorName` is set, `LookupByUpstream` iterates registered vendors looking for a name match and returns nil if none found (thirdparty/vendors_registry.go:L55-61). `NewUpstream` then proceeds with `vendor = nil`, skipping `GenerateConfigs` and error normalizer wiring entirely. No error is logged at creation time; the upstream runs as a bare HTTP endpoint without vendor-specific behavior. This can be a footgun when a vendorName typo is made.
